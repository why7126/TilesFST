# REQ-0065 瓷砖 SKU 元数据名称与编码展示去重需求文档

- 版本：v1
- 状态：approved
- 产品：菲尚特瓷砖信息管理平台
- 范围：管理端 SKU 管理、后端 SKU 主数据、小程序/店主端商品展示
- 更新日期：2026-07-21

## 1. 背景

当前瓷砖 SKU 主数据同时存在 SKU 名称与 SKU 编码。两者都在部分管理端和小程序/店主端场景中展示，容易让运营和终端用户感知为重复信息。

本需求不取消 SKU 编码的系统价值：SKU 编码仍作为唯一编码、系统识别字段、接口定位与内部排障依据。但 SKU 编码应由系统自动生成，不要求运营手工填写，也不默认呈现给小程序/店主端用户。用户侧主要展示由运营填写的 SKU 名称，并在管理端文案上明确为“商品名称”。

## 2. 目标

1. 降低管理端 SKU 维护成本，避免运营重复填写“名称”和“编码”两个相似字段。
2. 统一用户可见商品标识，小程序/店主端只展示商品名称，暂不展示 SKU 编码。
3. 保留 SKU 编码作为系统唯一识别字段，保证后台检索、接口兼容、数据追溯和故障排查能力。
4. 明确管理端字段语义：商品名称由用户填写并对外展示；SKU 编码由系统生成并用于内部识别。

## 3. 目标用户

| 用户 | 核心诉求 |
|---|---|
| 后台运营 | 创建 SKU 时只填写对外展示的商品名称，减少重复录入 |
| 后台管理员 | 仍可在后台识别、检索和排查具体 SKU |
| 店主端/小程序用户 | 浏览商品时只看到自然的商品名称，不被内部编码干扰 |
| 开发与运维 | 保持唯一编码用于 API、日志、数据迁移和问题定位 |

## 4. 范围

### 4.1 本期包含

- 管理端 SKU 新增/编辑表单字段文案与交互调整。
- SKU 编码由系统自动生成，运营创建 SKU 时无需手工填写。
- 管理端可在必要的内部管理场景查看或搜索 SKU 编码，但不把它作为用户主要录入字段。
- 小程序/店主端商品卡片、商品详情、搜索结果、推荐、分享标题等用户可见位置只展示商品名称，暂不展示 SKU 编码。
- 后端继续保证 SKU 编码唯一性，并保证历史数据可平滑兼容。
- 如 API 字段仍返回 `sku_code`，需明确其为内部字段，小程序/店主端不直接渲染。

### 4.2 本期不包含

- 不删除数据库中的 SKU 编码字段。
- 不将 SKU 编码开放为小程序/店主端展示卖点。
- 不引入手工编码规则配置、批量导入导出编码规则或外部 ERP 编码对接。
- 不新增商品别名、多语言商品名或系列命名体系。
- 不调整品牌、类目、规格、价格、图片、视频等非名称/编码字段。

## 5. 产品决策

1. `SKU 编码` 是系统唯一编码/系统识别字段，由系统自动生成。
2. `SKU 名称` 改以用户理解命名为 `商品名称`，由运营填写。
3. 小程序/店主端暂不展示 SKU 编码，只展示商品名称。
4. 管理端可以保留 SKU 编码的内部查看与搜索能力，但不要求运营在创建时填写。
5. 草稿与正式创建都应具备系统生成的唯一 SKU 编码，避免保存后再补齐编码造成数据不一致。

## 6. 功能要求

### FR-001 商品名称字段

- 管理端 SKU 新增/编辑表单将“SKU 名称”调整为“商品名称”。
- 商品名称为运营必填字段。
- 商品名称用于管理端列表主标题、小程序/店主端商品卡片、商品详情标题、推荐卡片和分享标题。
- 商品名称为空时不得创建正式 SKU；草稿是否允许为空需在后续 `/req-complete` 阶段结合现有草稿策略确认。

### FR-002 SKU 编码自动生成

- SKU 编码由系统自动生成，创建草稿和正式创建时均应生成唯一编码。
- 运营创建 SKU 时无需手工填写 SKU 编码。
- 系统生成的 SKU 编码必须唯一、稳定，创建后不得因商品名称修改而自动变化。
- SKU 编码仍可用于后台检索、接口定位、日志追踪和数据迁移。

### FR-003 管理端展示与检索

- 管理端 SKU 列表主展示使用商品名称。
- SKU 编码可作为内部辅助信息展示在管理端详情、调试信息或次级位置；若展示，视觉层级必须弱于商品名称。
- 管理端关键词搜索应继续支持商品名称与 SKU 编码，便于内部排障和精确定位。
- 上架、下架、删除确认文案使用商品名称，不使用 SKU 编码作为用户可读主标题。

### FR-004 小程序/店主端展示

- 小程序/店主端商品卡片只展示商品名称，不展示 SKU 编码。
- SKU 详情页标题、参数区、推荐卡片、同品牌/同系列商品、收藏列表和分享标题均不展示 SKU 编码。
- 搜索结果中 SKU 项使用商品名称作为主展示；是否仍支持用 SKU 编码搜索可作为内部兼容能力，但命中结果不展示编码。
- 分享标题使用品牌名称 + 商品名称，不拼接 SKU 编码。

### FR-005 API 与字段兼容

- 后端可以继续返回 `sku_code` 以支持管理端和内部兼容，但小程序/店主端前端不得直接渲染该字段。
- 若调整请求体，使 `sku_code` 不再由前端传入，必须同步 OpenAPI、Orval、接口文档和测试。
- 若保留前端类型中的 `sku_code`，需明确该字段是内部字段或兼容字段。

### FR-006 历史数据兼容

- 既有 SKU 的 `sku_code` 保持不变。
- 既有 SKU 的 `name` 作为商品名称继续展示。
- 若历史数据中名称与编码高度相似，不做自动清洗；本需求只调整后续展示与录入策略。
- 若历史数据缺少 SKU 编码，系统需补齐唯一编码后才能进入正式使用状态。

## 7. UI 约束

- 管理端字段 label 使用“商品名称”和“SKU 编码”，避免继续使用“SKU 名称”造成概念重复。
- 商品名称在列表和弹窗中保持主视觉层级。
- SKU 编码如在管理端展示，使用弱化文本，不与商品名称并列竞争。
- 小程序/店主端不出现“SKU 编码：xxx”参数行。
- 小程序/店主端商品标题、卡片标题和分享标题不得由 SKU 编码兜底，除非商品名称缺失且进入异常兼容状态。

## 8. 数据与接口影响

| 范围 | 影响 |
|---|---|
| SQLite/MySQL | 默认保留 `tiles.sku_code` 唯一字段；若改为后端自动生成，需确认创建链路与迁移脚本 |
| Pydantic Schema | 创建请求中 `sku_code` 可能从必填/可传改为后端生成或只读字段 |
| OpenAPI/Orval | 如请求/响应 contract 调整，必须同步生成与前端调用 |
| 管理端 Web | 表单、列表、搜索 placeholder、确认弹窗文案需调整 |
| 小程序/店主端 | 商品卡片、详情参数、搜索结果、分享标题需隐藏编码 |
| 测试 | 需覆盖自动编码、唯一性、展示隐藏、搜索兼容和历史数据 |
