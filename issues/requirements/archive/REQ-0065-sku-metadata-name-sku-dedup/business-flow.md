# 业务流程

## 主流程

```text
后台运营打开 SKU 新增/编辑弹窗
        ↓
填写商品名称、品牌、类目、规格、价格、媒体等业务字段
        ↓
系统在保存时生成或保持唯一 SKU 编码
        ↓
后端保存 SKU 主数据
  ├─ 商品名称：用户填写，对外展示
  └─ SKU 编码：系统生成，内部识别
        ↓
管理端列表刷新
  ├─ 主标题展示商品名称
  └─ SKU 编码仅作为内部辅助信息或搜索命中依据
        ↓
小程序/店主端消费公开商品数据
  ├─ 商品卡片展示商品名称
  ├─ SKU 详情展示商品名称和业务参数
  ├─ 搜索结果展示商品名称
  └─ 分享标题展示品牌名称 + 商品名称
```

## 管理端搜索流程

```text
管理员输入关键词
  ├─ 输入商品名称关键词
  └─ 输入已知 SKU 编码
        ↓
后端按商品名称 OR SKU 编码检索
        ↓
列表返回匹配 SKU
        ↓
页面主展示仍以商品名称为主，编码不抢占主标题
```

## 异常流程

```text
创建正式 SKU 时商品名称为空
  └─ 阻止提交，在商品名称字段下方展示校验错误

自动生成 SKU 编码冲突
  ├─ 后端重试生成或返回可理解错误
  └─ 不要求运营手动处理编码

历史 SKU 缺少编码
  ├─ 迁移或兼容逻辑补齐唯一编码
  └─ 补齐前不得进入正式公开链路

公开端收到 sku_code 字段
  └─ 前端不渲染，继续使用商品名称展示
```

## 与父需求差异

父需求 `REQ-0006-tile-sku-management` 定义 SKU 管理主能力，包括 SKU 名称、编码、品牌、类目、规格、价格、图片、视频和状态。本需求是该父需求的字段语义与展示策略 refinement：

- 不新增 SKU 管理页面。
- 不删除 SKU 编码底层字段。
- 不改变图片、视频、价格、品牌、类目和规格管理能力。
- 聚焦“商品名称由用户填写并展示、SKU 编码由系统生成并内部使用”的端到端一致性。

## 数据与接口边界

- SQLite/MySQL 默认继续保留 `tiles.sku_code` 唯一字段。
- 后端创建链路负责生成 SKU 编码；前端不再承担编码输入责任。
- 管理端 API 如调整请求体，必须同步 OpenAPI、Orval、docs 和测试。
- 小程序/店主端 API 可保留 `sku_code` 作为兼容字段，但公开端 UI 不渲染。

## UI 与原型策略

- 原型位于 `prototype/web/`，用于固定管理端列表和新增/编辑弹窗的字段语义与视觉层级。
- 小程序/店主端不单独生成原型，本需求通过 acceptance 明确隐藏 SKU 编码的展示规则。
- PNG Golden Reference 可在后续 OpenSpec Change 或实现阶段按需导出，不阻塞需求评审。
