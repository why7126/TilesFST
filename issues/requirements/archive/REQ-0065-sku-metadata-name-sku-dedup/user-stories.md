# 用户故事

## US-001 运营填写商品名称

作为后台运营，我希望创建或编辑 SKU 时只需要填写面向用户展示的商品名称，而不需要手工填写内部 SKU 编码，以便减少重复录入和命名困惑。

验收要点：

- 管理端新增/编辑弹窗将主输入字段命名为“商品名称”。
- 创建正式 SKU 时商品名称必填。
- SKU 编码由系统自动生成，不要求运营手工输入。
- 编辑商品名称不会导致已有 SKU 编码变化。

## US-002 管理员内部定位 SKU

作为后台管理员，我希望系统仍保留唯一 SKU 编码，并支持通过商品名称或 SKU 编码检索，以便进行排障、数据核对和精确定位。

验收要点：

- 每个 SKU 均有唯一、稳定的系统编码。
- 管理端关键词搜索支持商品名称和 SKU 编码。
- SKU 编码如在管理端展示，视觉层级弱于商品名称，不作为用户可读主标题。
- 上架、下架、删除等确认文案使用商品名称。

## US-003 小程序用户浏览商品

作为小程序用户，我希望商品卡片、详情页和推荐区域只展示自然的商品名称，而不是内部 SKU 编码，以便浏览体验更清晰。

验收要点：

- 商品卡片标题只展示商品名称。
- SKU 详情页标题、参数区和推荐卡片不展示 SKU 编码。
- 接口即使返回 `sku_code`，小程序端也不直接渲染该字段。
- 商品名称缺失属于异常数据，不用 SKU 编码作为正常兜底展示。

## US-004 店主端分享商品

作为门店导购或店主端使用者，我希望分享给客户的商品标题只包含品牌和商品名称，以便分享卡片更像商品信息而非内部编码。

验收要点：

- 分享标题使用品牌名称 + 商品名称。
- 分享摘要、卡片标题和路径展示不拼接 SKU 编码。
- 分享进入详情后，页面仍只展示商品名称。

## US-005 开发与运维保持兼容

作为开发或运维人员，我希望 SKU 编码继续作为内部识别字段存在，以便 API、日志、迁移和历史数据兼容不被破坏。

验收要点：

- 既有 SKU 编码保持不变。
- 新 SKU 自动生成编码，且唯一性由后端保证。
- 若 API 请求体不再接收前端传入 `sku_code`，需同步 OpenAPI、Orval、文档和测试。
- 历史数据名称与编码高度相似时不做自动清洗，只调整后续录入与展示策略。
