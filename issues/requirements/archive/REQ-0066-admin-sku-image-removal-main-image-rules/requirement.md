# REQ-0066 管理端 SKU 编辑弹窗商品图片移除与主图规则需求文档

- 版本：v1
- 状态：approved
- 产品：菲尚特瓷砖信息管理平台
- 范围：Web 管理端 SKU 编辑弹窗
- 更新日期：2026-07-22

## 1. 背景

`REQ-0006-tile-sku-management` 已定义瓷砖 SKU 管理的多图上传、主图指定和非主图删除能力。当前管理端 SKU 编辑弹窗中，商品图片可以继续添加和设置主图，但已添加图片缺少完整的移除操作；同时主图切换后不会自动移动到图片列表首位，移除当前主图后的兜底主图规则也未明确。

这会导致运营在维护 SKU 图片时无法闭环处理错误图片、过期图片或主图调整，保存后图片顺序与主图状态也可能不符合展示预期。REQ-0066 用于补齐 SKU 编辑弹窗中的图片移除、主图前置和主图兜底规则。

## 2. 目标

1. 支持运营在 SKU 编辑弹窗中移除任意已添加商品图片。
2. 明确设置主图后的排序规则：当前主图默认移动到第一个位置。
3. 明确移除当前主图后的兜底规则，避免 SKU 在仍有图片时意外缺少主图。
4. 保证保存后再次打开弹窗时，图片顺序、主图标记与列表素材状态一致。
5. 在不扩大媒体存储治理范围的前提下，先完成 SKU 与图片关联层面的编辑闭环。

## 3. 目标用户

| 用户 | 核心诉求 |
|---|---|
| 后台运营 | 编辑 SKU 时删除错误图片、替换主图，并确保展示首图符合运营意图 |
| 后台管理员 | 维护商品资料质量，减少错误图片残留和主图缺失 |
| 只读账号 | 不涉及写操作，仅在列表或详情中查看保存后的主图结果 |

## 4. 范围

### 4.1 本期包含

- 管理端 SKU 编辑弹窗的商品图片移除入口与交互。
- 设置主图后自动将该图片移动到图片列表第一个位置。
- 移除非主图图片时，当前主图保持不变。
- 移除当前主图时，如果仍有其他图片，自动选择兜底主图。
- 保存 SKU 后，图片顺序、`is_main` 和 `sort_order` 与弹窗展示一致。
- 前端组件测试覆盖图片移除、主图切换、主图兜底和全部移除场景。

### 4.2 本期不包含

- 不新增图片拖拽排序能力。
- 不新增批量删除图片能力。
- 不改变图片上传接口、上传格式、大小限制或 MinIO 前缀策略。
- 不做对象存储物理文件删除；本期仅解除 SKU 与图片的关联。
- 不调整小程序、店主端或 Web 展示端的商品图片展示规则。
- 不新增独立媒体资产回收、孤儿文件清理或审计能力。

## 5. 产品决策

1. 商品图片移除应面向所有图片，包括当前主图。
2. 主图在视觉和保存顺序上都应排在第一位，便于运营理解“第一张即主图”。
3. 移除当前主图时，以移除前的图片列表顺序为依据，优先选择当前主图后一张图片作为新主图。
4. 如果当前主图已经是最后一张，但移除后仍有其他图片，则选择移除后列表第一张图片作为新主图。
5. 如果移除后没有任何图片，则允许保存为缺图片/缺主图状态；是否允许正式上架由既有 SKU 上架策略约束。
6. 已上传但被移除的图片不在本需求中做对象存储物理删除，避免误删其他业务可能复用的媒体对象。

## 6. 功能要求

### FR-001 图片移除入口

- SKU 编辑弹窗中每张商品图片均应提供移除操作。
- 移除入口应位于图片缩略图卡片上，和“主图”“设为主图”状态区分清楚。
- 移除操作不得影响视频列表、SKU 基础字段、品牌类目字段或价格字段。
- 只读账号不得看到编辑弹窗或图片移除入口。

### FR-002 移除非主图图片

- 当用户移除非主图图片时，当前主图保持不变。
- 剩余图片的展示顺序应保持稳定，并重新生成连续的 `sort_order`。
- 保存后再次打开编辑弹窗，主图仍位于第一个位置，图片数量减少一张。

### FR-003 设置主图并前置

- 用户点击“设为主图”后，该图片立即成为唯一主图。
- 新主图应自动移动到图片列表第一个位置。
- 原主图取消主图标记，并保留在剩余图片列表中。
- 保存 payload 中应只有一张图片满足 `is_main=true`，且主图 `sort_order=0`。

### FR-004 移除当前主图后的兜底

- 当用户移除当前主图且当前主图后一张图片存在时，后一张图片应自动成为新主图。
- 当用户移除当前主图且当前主图后一张图片不存在，但仍有其他图片时，移除后列表第一张图片应自动成为新主图。
- 新主图应移动或保持在图片列表第一个位置。
- 兜底主图变化应在弹窗内即时可见，不等待保存后才体现。

### FR-005 移除全部图片

- 当最后一张图片被移除后，图片列表为空。
- 弹窗不再显示主图标记。
- 保存草稿时允许缺图片/缺主图，并沿用既有素材完整度状态。
- 如正式上架或创建动作存在主图要求，应复用既有校验与提示，不在本需求中新增新的上架策略。

### FR-006 保存与回填一致性

- 保存 SKU 时，前端提交的图片数组应反映弹窗中的最终图片列表。
- 被移除的图片不应出现在保存 payload 中。
- 图片数组应包含稳定的 `object_key`、`url`、`is_main` 与 `sort_order`。
- 后端保存后再次查询 SKU 详情，返回的图片列表顺序与主图标记应与保存前一致。

## 7. UI 约束

- 图片移除操作应使用明确的删除/移除图标或按钮，并提供可访问名称。
- “主图”标签继续保留在主图缩略图上，非主图保留“设为主图”操作。
- 移除按钮不得遮挡主图标签和图片主体识别。
- 空图片状态下继续展示“继续添加图片”入口和既有帮助文案。
- 交互风格沿用管理端 SKU 弹窗现有暗色旗舰风、品牌金强调、低圆角和语义化 token。
- 移除操作如增加确认，应保持轻量；除非后续评审明确要求，本需求不强制二次确认。

## 8. 数据与接口影响

| 范围 | 影响 |
|---|---|
| 管理端 Web | 需要调整 SKU 编辑弹窗图片状态管理、移除入口、主图前置与兜底算法 |
| 后端 API | 预计不新增接口；继续使用现有 SKU 更新接口提交 images 全量列表 |
| SQLite/MySQL | 预计不新增表字段；继续使用 `tile_images.is_main` 与 `sort_order` |
| Pydantic Schema | 预计不新增字段；需保持 images item 的 `object_key`、`url`、`is_main`、`sort_order` 语义 |
| OpenAPI/Orval | 若 API contract 不变，则不需要重新生成；若后续实现调整请求体再同步 |
| MinIO/对象存储 | 不做物理删除；仅解除 SKU 关联 |
| 测试 | 需补充前端组件测试；必要时补充后端保存顺序与主图唯一性的回归测试 |
