# 用户故事

## US-001 产品负责人创建产品版本发布计划

作为产品负责人，我希望能够以产品版本为单位创建发布计划，并将多个 Sprint 的交付内容合并到同一个版本中，以便对外公告与内部追踪使用同一个版本事实源。

验收要点：

- 能标识产品版本号，例如 `v0.1.0`。
- 能关联一个或多个 Sprint。
- 能从关联 Sprint 汇总 REQ、BUG 和 OpenSpec Change。
- 未评审或未归档的内容不得进入正式发布范围。

## US-002 开发与测试执行发布前校验

作为开发或测试，我希望发布前有固定校验清单，以便在发布公告生成前确认 OpenSpec、测试、Orval、Docker、数据库迁移和环境变量同步均已完成。

验收要点：

- 校验失败时必须阻断发布流程。
- 校验结果必须写入发布材料或 trace。
- 涉及 API、数据库、Docker、环境变量时必须提示对应文档和生成物同步。

## US-003 运维和实施查看升级与回滚说明

作为运维或实施人员，我希望每个产品版本公告都包含升级步骤、回滚说明、已知问题和影响范围，以便上线前后能执行明确的操作计划。

验收要点：

- 公告必须包含升级步骤。
- 公告必须包含回滚说明。
- 公告必须包含已知问题和影响范围。
- 公告不得包含敏感凭据、真实客户数据或内部连接串。

## US-004 公开访客查看版本公告

作为客户、店主或公开访客，我希望通过公开页面查看产品版本公告，以便了解当前版本新增功能和修复内容。

验收要点：

- 公告以 Mintlify 静态文档生成。
- 公告可以在公开页面阅读。
- 公告内容语言清晰，面向公开读者。
- 公告展示不依赖后端运行时 API 或数据库。

## US-005 AI Agent 按命令生成发布材料

作为 AI Agent，我希望有明确的发布命令和目录规范，以便稳定生成产品版本发布对象、Mintlify 公告和校验报告。

验收要点：

- 后续命令族必须从 `.cursor/commands/` 同步到其他 Agent 工具。
- 新增顶层 `releases/` 前必须先通过 OpenSpec Change 修改目录规范。
- 发布材料必须可追踪到 Sprint、REQ、BUG 和 OpenSpec Change。
