# REQ-0062 管理后台 Banner 投放范围配置优化

## 1. 需求背景

`REQ-0016-banner-management` 已交付管理后台 Banner 管理基础能力，当前模型和页面支持 Web 首页、小程序首页、专题页等多个展示端与展示位置。随着微信小程序首页和品牌列表页逐步成为当前主要展示场景，运营只需要维护小程序内的两个轮播位：小程序首页轮播与小程序品牌列表页轮播。

现有 Banner 管理范围过宽，会让运营在创建和筛选 Banner 时看到 Web 首页、专题页、首页中部运营位等当前不再需要的投放目标。品牌列表页需求 `REQ-0060-brand-list-page` 也需要独立的品牌列表页轮播数据来源，不能长期复用首页轮播位造成内容混淆。

本需求将 Banner 管理收敛为小程序投放能力，并明确旧数据处理策略：历史非小程序展示端和非目标展示位置的数据在后续实现中删除，不做长期兼容保留。

## 2. 目标用户

| 用户 | 诉求 |
|---|---|
| 后台管理员 | 只维护当前有效的小程序 Banner 投放位，减少误配置。 |
| 后台运营 | 能分别配置小程序首页轮播和品牌列表页轮播，避免两个页面共用同一批 Banner。 |
| 小程序访客 | 在首页和品牌列表页看到与页面语境一致的轮播内容。 |

## 3. 需求目标

1. 管理后台 Banner 展示端仅保留“小程序”。
2. Banner 展示位置仅保留“首页轮播”和“品牌列表页轮播”。
3. 新建和编辑 Banner 时只能选择上述合法组合。
4. 小程序首页与品牌列表页分别读取对应轮播位，不再让品牌列表页复用首页轮播数据。
5. 删除历史旧数据，避免 Web 首页、专题页或旧运营位 Banner 继续污染列表、统计和前台查询。

## 4. 范围

### 4.1 本期包含

| 范围 | 说明 |
|---|---|
| 管理端展示端收敛 | Banner 列表筛选、新增/编辑弹窗、展示文案仅保留“小程序”。 |
| 管理端展示位置收敛 | 展示位置仅包含“首页轮播”和“品牌列表页轮播”。 |
| 后端枚举校验 | 后端只接受小程序展示端及两个展示位置。 |
| 旧数据删除 | 删除历史 Web 首页、专题页、首页中部运营位等不在新范围内的 Banner 数据。 |
| 小程序查询分流 | 小程序首页读取首页轮播；品牌列表页读取品牌列表页轮播。 |
| API / OpenAPI / Orval 同步 | 若请求、响应或枚举发生变化，必须同步 OpenAPI、Orval 和接口文档。 |
| 测试覆盖 | 覆盖后端校验、旧数据清理、管理端选项、小程序两个轮播位查询。 |
| 品牌详情跳转 | Banner 跳转类型新增“品牌详情”，整体交互参照 SKU 详情：选择品牌、可使用品牌 Logo 作为 Banner 图、点击后进入品牌详情页。 |

### 4.2 本期不包含

| 不包含 | 说明 |
|---|---|
| 新增 Banner 表 | 继续使用现有 Banner 数据模型，按枚举和数据清理收敛范围。 |
| Web 首页 Banner 展示 | Web 展示端 Banner 不再属于本需求范围。 |
| 专题页 Banner | 专题页 Banner 数据和配置入口删除，不做新专题轮播维护。 |
| 多租户投放位 | 不新增店铺级、品牌级或区域级投放位。 |
| 点击统计改造 | Banner 点击埋点如已有则沿用；本需求不新增统计报表。 |
| 拖拽排序 | 继续使用排序字段，不新增拖拽排序能力。 |

## 5. 术语与枚举

### 5.1 展示端

| 中文 | 建议枚举 | 说明 |
|---|---|---|
| 小程序 | `MINIAPP` 或沿用兼容枚举 `MINIAPP_HOME` | 后续 OpenSpec 阶段确定最终枚举命名；管理端文案必须显示为“小程序”。 |

旧展示端 `WEB_HOME`、`TOPIC` 不再可创建、编辑或查询为有效业务范围。

### 5.2 展示位置

| 中文 | 建议枚举 | 说明 |
|---|---|---|
| 首页轮播 | `MINIAPP_HOME_CAROUSEL` | 小程序首页顶部轮播。 |
| 品牌列表页轮播 | `MINIAPP_BRAND_LIST_CAROUSEL` | 小程序品牌列表页顶部轮播。 |

旧展示位置 `HOME_TOP_CAROUSEL`、`HOME_MID_SLOT`、`TOPIC_TOP_BANNER` 以及其他不在上表中的值，后续实现时应清理删除。

## 6. 功能要求

### FR-001 管理端展示端收敛

- Banner 管理列表筛选 MUST 只展示“小程序”一个展示端选项。
- 新增 Banner 弹窗的展示端 MUST 默认且固定为“小程序”。
- 编辑 Banner 弹窗中展示端 MUST 为“小程序”，不得编辑为 Web 首页、专题页或其他端。
- 如 UI 仍保留展示端控件，控件 MUST 表达单一选项或只读状态；不得出现无效多选项。
- 列表表格展示端列 MUST 展示“小程序”，不得展示旧端文案。

### FR-002 管理端展示位置收敛

- Banner 管理列表、筛选和弹窗中的展示位置 MUST 仅包含“首页轮播”和“品牌列表页轮播”。
- 新建 Banner 默认展示位置 SHOULD 为“首页轮播”。
- 用户切换展示位置时，其他字段不应被无故清空；跳转类型切换仍按既有规则清理不兼容字段。
- 列表表格 MUST 保留“展示位置”独立列，并展示上述中文文案。
- 无效旧位置不得作为可选项出现。

### FR-003 后端合法组合校验

- 服务端创建和更新 Banner 时 MUST 只接受“小程序 + 首页轮播”与“小程序 + 品牌列表页轮播”两个组合。
- 当请求包含旧展示端或旧展示位置时，服务端 MUST 返回统一校验错误，不得静默保存。
- 列表 API 默认 MUST 只返回新范围内的 Banner。
- 如果实现阶段保留 `display_client` 字段，服务端 MUST 保证它不会再接受非小程序值。
- 如果实现阶段新增 `position` 枚举，数据库 schema、Pydantic Schema、OpenAPI 和前端类型 MUST 同步。

### FR-004 旧数据删除

- 实现阶段 MUST 删除不在新范围内的历史 Banner 数据，包括但不限于：
  - `display_client=WEB_HOME`
  - `display_client=TOPIC`
  - `position=HOME_TOP_CAROUSEL`
  - `position=HOME_MID_SLOT`
  - `position=TOPIC_TOP_BANNER`
  - 其他无法映射到“首页轮播”或“品牌列表页轮播”的旧值
- 删除旧数据前 MAY 在迁移日志或执行输出中记录删除数量和条件，便于验收追踪。
- 删除后列表统计、分页、筛选和 summary MUST 不再包含旧数据。
- 旧数据删除只针对 Banner 业务记录；不得删除 MinIO 中仍被 SKU、品牌或其他业务引用的媒体对象。
- 如 Banner 图片对象仅被被删除 Banner 引用，是否物理清理对象存储可作为后续运维任务，不作为本需求必做项。

### FR-005 小程序首页轮播查询

- 小程序首页 MUST 只读取“首页轮播”位置的在线且有效 Banner。
- 首页轮播排序沿用 `sort_order ASC, updated_at DESC` 或现有稳定排序策略。
- 首页轮播不应读取品牌列表页轮播数据。
- 无首页轮播数据时，首页按已有降级策略展示，不得白屏。

### FR-006 小程序品牌列表页轮播查询

- 小程序品牌列表页 MUST 只读取“品牌列表页轮播”位置的在线且有效 Banner。
- 品牌列表页轮播可复用首页轮播组件的视觉和交互，但数据源 MUST 与首页轮播区分。
- 无品牌列表页轮播数据时，品牌列表页 SHOULD 隐藏轮播或展示品牌化兜底，不得展示首页轮播作为兜底。
- 点击跳转、图片安全 URL、有效期过滤和排序规则 SHOULD 与首页轮播保持一致。

### FR-007 管理端新增/编辑弹窗

- 新增弹窗打开时，展示端 MUST 为“小程序”，展示位置默认“首页轮播”。
- 编辑旧 Banner 数据的入口不应存在；旧数据已删除后，编辑弹窗只处理新范围内 Banner。
- Banner 图片、跳转类型、排序、有效期、运营备注等字段沿用 `REQ-0016-banner-management` 已定义规则。
- 弹窗保存时 MUST 使用后端合法组合校验结果，不得只依赖前端选项限制。

### FR-010 品牌详情跳转

- Banner 跳转类型 MUST 新增“品牌详情”。
- 选择“品牌详情”后，管理端 MUST 展示“关联品牌”可搜索选择控件，整体交互参照“SKU 详情”的关联 SKU 控件。
- 选择品牌后，运营 SHOULD 可一键使用品牌 Logo 作为 Banner 图；品牌无 Logo 时 MUST 提示自定义上传。
- 保存 `jump_type=BRAND_DETAIL` 时，服务端 MUST 要求 `brand_id` 非空且品牌存在并已启用。
- `BRAND_DETAIL` 不得同时携带 `sku_id`、`topic_id` 或 `external_url`。
- 小程序公开 Banner 响应 SHOULD 将品牌详情跳转表达为 `jump_type=brand` 与 `target_id=brand_id`，点击后进入品牌详情页。

### FR-008 列表和统计

- Banner 总数、当前筛选、已上线、待生效等指标卡 MUST 仅统计新范围数据。
- 列表关键词、状态、时间状态筛选继续沿用既有逻辑。
- 若保留展示端筛选控件，由于只有“小程序”，筛选结果应与默认列表一致。
- 分页在旧数据删除后 MUST 使用清理后的总数。

### FR-009 OpenAPI / Orval / 文档同步

- 任何枚举字段、请求体、响应体或错误码变化 MUST 同步 OpenAPI。
- Web 管理端 API 客户端类型 MUST 通过 Orval 重新生成，除非实现阶段证明生成物未变化。
- API 文档中 Banner 展示端与展示位置说明 MUST 与本需求一致。
- 数据库文档 MUST 更新 Banner 展示端、展示位置和旧数据删除策略。

## 7. UI / UE 约束

- 管理端仍遵守“工业石材 · 暗色旗舰风”和现有 Banner 管理页视觉基准。
- 不得因为选项减少破坏列表布局；表格列宽、筛选区域、弹窗对齐仍需保持管理端列表页一致性。
- 单选项控件应避免制造困惑：可使用只读文本、禁用选择器或单项下拉，但必须清楚表达当前只支持“小程序”。
- 展示位置两个选项的文案必须短且稳定：`首页轮播`、`品牌列表页轮播`。
- 错误提示应直接说明“当前仅支持小程序首页轮播和品牌列表页轮播”，避免暴露内部枚举名。

## 8. 数据处理策略

| 数据类型 | 策略 |
|---|---|
| 新范围内 Banner | 保留。 |
| 可明确映射到小程序首页轮播的历史小程序首页 Banner | 保留或保持为“首页轮播”。 |
| Web 首页 Banner | 删除。 |
| 专题页 Banner | 删除。 |
| 首页中部运营位 | 删除。 |
| 其他未知展示端/位置 | 删除。 |
| Banner 图片对象 | 不因删除 Banner 记录自动物理删除，避免误删共享素材；对象存储清理由后续运维策略处理。 |
| 品牌详情跳转 Banner | 新增 `brand_id` 作为目标；可使用品牌 `logo_object_key` 或自定义上传图片。 |

## 9. 权限与安全

- `admin` 与 `employee` 继续可访问 Banner 管理页面与 API。
- `store_owner` MUST NOT 访问 Banner 管理页面与 API。
- Banner 图片上传仍必须走后端鉴权、MIME/大小校验和 MinIO 适配层。
- 小程序公开接口只返回已上线、有效期内且位置匹配的 Banner，不暴露管理端备注、内部对象 key 或未授权字段。
- 旧数据删除不得绕过迁移审计；执行条件必须可复核。

## 11. 风险与待确认

| 风险 / 待确认 | 说明 |
|---|---|
| 枚举命名 | “展示端只有小程序”可实现为新枚举 `MINIAPP`，也可兼容沿用 `MINIAPP_HOME`；OpenSpec 阶段需最终确定。 |
| 历史数据删除 | 用户已明确“旧数据删除”；实现时仍需确保删除条件只作用于 Banner 业务记录，不误删媒体对象或其他业务数据。 |
| 品牌列表页接口 | 当前品牌列表页曾复用首页轮播数据；后续实现必须拆分品牌列表页轮播查询。 |
| 数据库约束 | SQLite/MySQL schema、Pydantic 校验和前端选项必须一致，否则会出现前端可选但后端拒绝或反向不一致。 |
| Orval 生成 | API 枚举变更通常需要重新生成前端 API 类型，需纳入实现任务。 |
| 品牌 Logo 缺失 | 品牌详情 Banner 若使用品牌 Logo 取图，需要品牌已有 Logo；无 Logo 时必须允许自定义上传作为运营兜底。 |
