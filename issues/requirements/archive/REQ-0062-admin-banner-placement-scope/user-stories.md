# REQ-0062 用户故事

## US-001 运营只看到小程序投放端

作为后台运营，我希望 Banner 管理中的展示端只有“小程序”，以便我不会误把 Banner 配置到 Web 首页或专题页等已不使用的投放目标。

验收要点：

- Banner 列表筛选、新增弹窗、编辑弹窗均不出现 Web 首页、专题页等旧展示端。
- 新增 Banner 时展示端默认为“小程序”。
- 旧展示端数据删除后，不再进入列表、统计和分页。

## US-002 运营区分首页和品牌列表页轮播

作为后台运营，我希望展示位置只有“首页轮播”和“品牌列表页轮播”，以便为首页和品牌列表页分别维护不同内容。

验收要点：

- 展示位置下拉或等价控件只出现两项。
- 首页轮播与品牌列表页轮播使用独立数据，不互相兜底污染。
- 列表中的展示位置列展示中文文案，便于运营扫描。

## US-003 管理员清理旧 Banner 数据

作为后台管理员，我希望旧 Banner 数据被删除，以便历史 Web 首页、专题页和旧运营位数据不再影响当前小程序投放范围。

验收要点：

- 迁移或清理逻辑删除不在新范围内的 Banner 记录。
- 删除条件和删除数量可复核。
- 删除 Banner 业务记录时不误删仍可能被其他业务引用的媒体对象。

## US-004 小程序访客看到页面语境一致的轮播

作为小程序访客，我希望首页和品牌列表页分别展示符合页面语境的轮播，以便首页营销和品牌频道内容不会混用。

验收要点：

- 小程序首页只读取首页轮播数据。
- 品牌列表页只读取品牌列表页轮播数据。
- 两个页面均只展示已上线、有效期内、图片 URL 可访问的 Banner。

## US-005 开发者获得清晰的 API 和数据边界

作为开发者，我希望 PRD 明确 API、数据库、Orval 和测试同步要求，以便实现时不会出现前端选项、后端校验、数据库约束和小程序查询不一致。

验收要点：

- 枚举或 Schema 变更同步 OpenAPI、Orval 和接口文档。
- SQLite/MySQL schema 与 Pydantic 校验保持一致。
- 后端测试覆盖非法旧枚举拒绝、旧数据删除和两个小程序位置查询。

## US-006 运营配置品牌详情 Banner

作为后台运营，我希望 Banner 跳转类型可以选择“品牌详情”，并像配置 SKU 详情一样搜索选择品牌，以便品牌列表页或首页 Banner 可以直接引导访客进入品牌主页。

验收要点：

- 跳转类型包含“品牌详情”。
- 选择“品牌详情”后展示关联品牌可搜索选择控件。
- 可使用品牌 Logo 作为 Banner 图，也可自定义上传运营图。
- 小程序点击该 Banner 后进入对应品牌详情页。
