## US-001 管理普通用户时明确识别保底账号不可操作

**作为** 后台管理员，  
**我希望** 在用户管理列表中看到 `ADMIN_USERNAME` 对应账号的操作按钮为不可用状态，  
**以便** 我知道该账号是系统保底账号，不能误编辑、误重置密码或误停用。

### 验收要点

- 用户列表和用户详情返回 `is_protected` 与 `protected_reason`。
- 受保护账号行的「编辑」「重置密码」「冻结 / 解冻」「删除」均置灰。
- 置灰按钮提供明确原因提示，前端不硬编码 `admin`。
- 普通 `role=admin` 用户不受影响，仍可按原用户管理规则维护。

### 关联功能

- FR-001、FR-002、FR-008（见 `requirement.md`）

## US-002 后端强制保护受保护账号

**作为** 系统安全策略，  
**我希望** 所有管理端账号维护接口都在后端识别并拒绝破坏受保护账号的操作，  
**以便** 即使前端被绕过，保底管理员账号仍不会被修改、停用或删除。

### 验收要点

- `PATCH /api/v1/admin/users/{id}` 对受保护账号返回明确错误，资料不变。
- `POST /api/v1/admin/users/{id}/reset-password` 对受保护账号返回明确错误，`password_hash` 不变。
- `PATCH /api/v1/admin/users/{id}/status` 对受保护账号返回明确错误，`status` 不变。
- 错误响应保持统一 `ApiResponse` 结构，并登记错误码。

### 关联功能

- FR-003、FR-004、FR-005、FR-007

## US-003 受保护账号本人不能通过管理端改掉保底密码

**作为** 受保护超级管理员本人，  
**我希望** 系统在管理端个人改密入口阻止我修改保底账号密码，  
**以便** 保底账号的密码恢复边界由部署配置和运维流程控制，而不是被管理端交互误改。

### 验收要点

- 当前登录用户为 `ADMIN_USERNAME` 对应账号时，个人修改密码 API 按评审结论拒绝。
- 拒绝时不更新 `password_hash`，不递增 `token_version`。
- 前端展示接口返回的中文原因。
- 若产品评审决定允许本人改密，本故事须改为非本期或调整为“仅禁止他人重置”。

### 关联功能

- FR-006、FR-009

## US-004 保留运维级恢复能力

**作为** 运维 / 部署负责人，  
**我希望** 管理端保护策略不影响 `.env` 级启动恢复机制，  
**以便** 在演示、生产或紧急恢复场景中仍可通过配置恢复保底管理员账号。

### 验收要点

- 本需求不新增 `super_admin` 角色，不改变 `ADMIN_USERNAME` 配置方式。
- `ADMIN_RESET_PASSWORD_ON_STARTUP` 等运维级恢复能力不属于管理端用户操作。
- 文档和验收需明确管理端保护与部署恢复的边界。

### 关联功能

- FR-009
