# 用户故事

## US-001 系统管理员查看日志审计列表

作为系统管理员，我希望在管理端 SYSTEM 分组进入日志审计页，按日志类型、时间、操作者、路径、状态码和 request_id 查询日志，以便快速定位接口异常和关键操作。

验收要点：

- MUST 仅系统管理员可访问日志列表。
- MUST 支持分页、筛选、按时间倒序展示。
- MUST 能区分请求日志、行为事件与审计操作。
- MUST 提供 request_id 展示与复制入口。

## US-002 系统管理员查看日志详情

作为系统管理员，我希望打开单条日志详情，查看请求摘要、行为上下文、错误摘要和 metadata，以便完成排障和审计取证。

验收要点：

- MUST 使用抽屉或等价详情承载，不跳出当前筛选结果。
- MUST 展示基础信息、请求信息、行为上下文、错误摘要、metadata 分组。
- MUST 对敏感字段脱敏或不展示。
- MUST 在 metadata 解析失败时仍能展示基础字段。

## US-003 开发 / 运维人员通过 request_id 排障

作为开发或运维人员，我希望通过 request_id 串联一次请求的日志、错误和行为上下文，以便复现问题并定位失败环节。

验收要点：

- MUST 在 API 请求入口生成或透传 request_id。
- MUST 在日志列表和详情中展示 request_id。
- SHOULD 支持按 request_id 精确筛选。
- SHOULD 在错误摘要中展示 error_code、状态码和耗时。

## US-004 产品负责人查看核心行为事件

作为产品负责人，我希望平台按事件字典采集页面访问、搜索筛选、实体操作、媒体上传和登录等核心行为，以便理解功能使用情况。

验收要点：

- MUST 先通过 `tracking-events.md` 人为定义事件名、触发时机和属性。
- MUST 禁止随意拼接未登记的 event_name。
- MUST 由系统自动补充用户、角色、客户端、页面路径、request_id 和时间戳等上下文。
- SHOULD 将店主 Web 展示端与微信小程序覆盖范围作为 Sprint 评审确认项。

## US-005 研发 / 测试维护事件字典与上报契约

作为研发和测试人员，我希望事件字典能明确必填属性、可选属性、禁止属性和校验规则，以便前后端实现一致且可测试。

验收要点：

- MUST 将稳定事件名同步为代码枚举、Schema 或等价校验。
- MUST 校验必填属性、字段类型、字段长度和禁止字段。
- MUST 覆盖上报成功、上报失败不阻断主流程、非法事件被拒绝等测试。
- MUST 在新增事件前先更新事件字典、OpenSpec 与测试。

## US-006 非系统管理员不可访问日志审计能力

作为普通员工、店主或匿名用户，我不应看到日志列表和日志详情，以避免敏感信息泄露。

验收要点：

- MUST 对列表接口、详情接口和管理端路由做权限校验。
- MUST 对直链访问返回 403 或管理端无权限页。
- MUST 不向店主 Web 展示端、微信小程序或匿名端暴露管理端日志详情。

## US-007 后端自动记录 API 请求日志

作为平台后端，我需要在统一中间件或等价入口自动记录 API 请求摘要，以避免业务代码重复散落日志逻辑。

验收要点：

- MUST 记录 method、path、status_code、duration_ms、actor、client_type、request_id、created_at。
- MUST 支持异常请求错误摘要。
- MUST 默认排除健康检查、静态资源、Swagger 文档、媒体直出等噪声路由。
- MUST 使用参数化查询和 Repository 或统一数据访问层写入关系型数据库。
