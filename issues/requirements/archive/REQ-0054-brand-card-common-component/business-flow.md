# 微信小程序品牌卡片组件业务流程

## 1. 主流程

```text
用户打开 SKU 详情页
        |
        v
页面加载 SKU 详情数据
        |
        v
页面从 SKU 数据中提取 brand 展示对象
        |
        v
渲染 <brand-card>
        |
        +-- Logo 正常：展示 Logo
        |
        +-- Logo 缺失/失败：展示品牌首字或默认占位
        |
        v
用户点击品牌卡片
        |
        +-- 有 brand_entry_path：跳转配置入口
        |
        +-- 无 brand_entry_path 但有品牌名：跳转搜索页 keyword=品牌名
        |
        +-- 品牌信息不足：提示“品牌内容暂不可查看”
```

## 2. 异常流程

```text
品牌数据缺失
  -> 卡片可隐藏或展示不可用态
  -> 不影响 SKU 详情页其他模块

品牌名称缺失
  -> 展示“品牌信息待完善”
  -> 禁用搜索 fallback
  -> 可上报数据异常

Logo 加载失败
  -> 切换到占位图或品牌首字
  -> 记录 brand_card_image_failed
  -> 保持卡片布局与点击能力

入口跳转失败
  -> 尝试品牌名搜索 fallback
  -> fallback 仍失败时 toast 提示
  -> 记录 brand_card_unavailable_click
```

## 3. 与父 REQ 差异

| 项目 | REQ-0005-brand-management | REQ-0054-brand-card-common-component |
|---|---|---|
| 端 | Web 管理端 | 微信小程序 |
| 目标 | 品牌主数据维护 | 品牌信息展示组件化 |
| 核心对象 | 品牌列表、表单、状态、删除规则、Logo 上传 | 单个品牌卡片、Logo fallback、跳转入口、埋点 |
| 数据变更 | 涉及品牌 API、上传、权限与主数据 | 首版不新增 API/DB/上传策略 |
| 用户 | 企业内部管理员/员工 | 装修客户、设计师、门店导购、访客 |

## 4. 组件边界

```text
页面容器
  - 负责 SKU 详情数据加载
  - 负责页面加载态/错误态
  - 负责是否展示品牌卡片
  - 传入 brand、source 和事件回调

品牌卡片组件
  - 负责 Logo/名称/副文案展示
  - 负责占位与不可用态
  - 负责点击反馈和防重复点击
  - 触发跳转/埋点回调或使用传入入口

后续品牌页/列表
  - 负责品牌详情、品牌商品列表、分页和筛选
  - 不由本组件承接
```

## 5. 复盘风险承接

最近一期 `sprint-008` 复盘指出小程序页面链路多、设备/视口验收证据不足、组件边界容易重叠。本需求在后续实现和验收时应：

- 明确品牌卡片只负责单品牌展示，不承接品牌商品列表状态机。
- 对 320/375/430 pt 视口进行截图验收，确认卡片不撑破、不遮挡。
- 新增小程序组件后同步 `.ts` / `.js` 运行入口策略，避免开发源码与微信开发者工具实际加载文件漂移。
