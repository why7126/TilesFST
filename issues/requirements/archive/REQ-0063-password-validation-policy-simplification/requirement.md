# REQ-0063 密码校验规则简化

## 1. 需求背景

当前系统密码校验规则较复杂，历史修改密码需求中曾约束新密码为 8-32 位、至少包含字母和数字，并在部分入口叠加弱密码、错误提示与限流逻辑。本需求将密码校验策略简化为用户更容易理解和通过的基础规则：长度 5-32 位，同时包含英文字符和数字。

该需求是 `REQ-0015-password-change` 的策略调整，不直接改变认证权限边界，也不引入新的登录、找回密码或多因素认证流程。后续实现时必须保证后端校验、管理端表单提示、API 错误信息和测试用例一致，避免出现前端放行但后端拒绝，或后端规则已调整但页面仍提示旧规则的问题。

## 2. 目标用户

| 角色 | 诉求 |
|---|---|
| 管理端管理员 | 创建、重置或修改密码时使用更清晰、更宽松的密码规则 |
| 管理端运营 / 内部员工 | 修改本人密码时能快速理解密码要求并通过校验 |
| 系统维护人员 | 通过统一密码校验函数维护规则，减少多入口规则漂移 |

## 3. 范围

### 3.1 本期包含

- 后端统一密码校验规则调整为：5-32 位、包含英文字符、包含数字。
- 管理端涉及密码设置的入口同步提示新规则。
- API 校验失败 message 同步为新规则文案。
- 新规则覆盖修改本人密码、创建用户初始密码、管理员重置密码等需要设置新密码的入口。
- 相关单元测试 / 集成测试覆盖新旧边界值。

### 3.2 本期不包含

- 忘记密码、短信验证码、邮箱验证、多因素认证。
- 登录密码校验流程本身的鉴权逻辑调整。
- 历史已设置密码的批量重置或强制升级。
- 密码历史不可重复、密码有效期、密码强度评分等增强策略。
- 独立 UI 原型或新页面。

## 4. 功能要求

### FR-001 密码长度规则

系统设置新密码时，密码长度 MUST 满足 5-32 位。

| 场景 | 结果 |
|---|---|
| 长度少于 5 位 | 校验失败 |
| 长度等于 5 位 | 继续检查字符组成 |
| 长度等于 32 位 | 继续检查字符组成 |
| 长度超过 32 位 | 校验失败 |

### FR-002 英文字符规则

密码 MUST 至少包含 1 个英文字符。

- 英文字符默认按 ASCII 字母 `A-Z` / `a-z` 判断。
- 是否接受非 ASCII 拉丁扩展字符待后续评审确认；未确认前实现 SHOULD 仅按 ASCII 英文字母校验。

### FR-003 数字规则

密码 MUST 至少包含 1 个数字字符。

- 数字默认按 `0-9` 判断。
- 是否接受全角数字或其他 Unicode 数字待后续评审确认；未确认前实现 SHOULD 仅按 ASCII 数字校验。

### FR-004 其他字符处理

在满足长度、英文字符、数字三类规则的前提下，系统 MAY 允许符号等其他字符作为密码内容。

- 是否允许空格、中文字符等非英文/数字字符待后续评审确认。
- 后续实现不得因为“包含其他字符”而自动判定失败，除非 OpenSpec Change 明确收紧允许字符集。

### FR-005 适用入口

本需求的密码校验规则 MUST 覆盖所有“设置新密码”的入口：

| 入口 | 要求 |
|---|---|
| 管理端修改本人密码 | 新密码使用本规则 |
| 管理员创建用户 | 初始密码如由用户输入，使用本规则 |
| 管理员重置用户密码 | 重置密码如由用户输入或展示给用户，使用本规则 |
| 初始化管理员密码 | 若项目存在初始化密码校验，使用本规则 |

如果某入口当前由系统自动生成随机密码，生成结果也 SHOULD 满足本规则。

### FR-006 前后端提示一致性

管理端表单和后端 API 错误提示 MUST 使用一致的新规则说明。

推荐提示文案：

```text
密码需为 5-32 位，并同时包含英文字符和数字。
```

字段级提示 SHOULD 能区分以下原因：

| 场景 | 推荐提示 |
|---|---|
| 长度少于 5 位 | 密码至少需要 5 位 |
| 长度超过 32 位 | 密码至多 32 位 |
| 缺少英文字符 | 密码需包含英文字符 |
| 缺少数字 | 密码需包含数字 |

### FR-007 后端校验统一

后端 MUST 使用统一密码校验函数或等价公共逻辑承载该策略，避免创建用户、重置密码、修改密码等入口分别维护不同正则或条件判断。

如项目已有 `validate_password()` 或等价函数，后续实现 SHOULD 优先调整该函数，并让各入口复用。

### FR-008 兼容已有安全规则

本需求仅简化基础复杂度规则，不默认移除以下既有安全约束：

- 修改本人密码仍需校验原密码。
- 新旧密码是否允许相同按现有修改密码需求继续执行，除非后续 OpenSpec 明确调整。
- 修改密码限流、弱密码表、token 失效等已交付安全能力不因本需求自动取消。

## 5. UI 约束

- 管理端密码输入组件应继续复用现有 Design System 与业务组件，不新增独立页面样式。
- 规则提示文案应放在密码输入框附近，避免只依赖提交后的全局 Toast。
- 错误态应使用既有表单错误样式和 semantic token。
- 页面中不得继续展示“至少 8 位”等旧规则文案。
