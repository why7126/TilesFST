# 用户故事

## US-001 后台运营新增类目时无需填写内部编码

**作为** 后台管理员或内部员工，  
**我希望** 新增类目时只填写业务可理解字段，不需要看到或填写类目编码，  
**以便** 降低维护成本，并避免人为输入不规范编码。

验收要点：

- 新增类目弹窗不展示类目编码输入项。
- 保存请求不依赖前端提交 `code`。
- 创建成功后的类目仍有系统生成的 `CAT-` 前缀唯一编码。
- 生成失败或冲突时返回稳定业务错误，不暴露数据库原始异常。

## US-002 后台运营通过必填标识快速识别必填字段

**作为** 后台管理员或内部员工，  
**我希望** 上级类目、类目名称、排序权重都有一致的必填标识，  
**以便** 在打开弹窗时即可知道哪些字段必须填写。

验收要点：

- 上级类目、类目名称、排序权重标签附近展示必填标识。
- 创建一级类目时，上级类目通过「无上级，创建一级类目」表达。
- 排序权重为空、0、负数、小数或非数字时阻止保存。
- 字段级错误展示在对应字段或字段组下方。

## US-003 后台运营维护类目名称时得到即时且一致的校验

**作为** 后台管理员或内部员工，  
**我希望** 类目名称只能使用短且规范的名称，  
**以便** 保持后台类目树、小程序分类页和 SKU 挂载数据清晰一致。

验收要点：

- 类目名称 trim 后不能为空。
- 类目名称最多 10 个字符。
- 类目名称仅允许中文、英文和数字。
- 非法输入在前端保存前提示，后端 API 仍会兜底拒绝。

## US-004 后台运营不能在同一层级创建重复类目

**作为** 后台管理员或内部员工，  
**我希望** 同一上级类目下不能存在重名类目，  
**以便** 避免商品挂载、前台展示和运营检索出现歧义。

验收要点：

- 同一 `parent_id` 下创建重名类目时保存失败。
- 顶级类目之间按 `parent_id = null` 做重名校验。
- 编辑类目保持原名称时不误判自身重复。
- 编辑类目改名为同层级其他类目名称时保存失败。

## US-005 后续开发者能够从需求包直接识别 API 和 Orval 影响

**作为** 后续实现该需求的开发者，  
**我希望** 需求包清楚指出创建 API 不再要求 `code` 且需要同步 OpenAPI / Orval，  
**以便** 后续 Change 不遗漏接口契约、错误码和测试更新。

验收要点：

- PRD、验收标准和 trace 均标明 API / Orval 影响。
- 后端 Pydantic Schema、OpenAPI、前端 API 客户端和测试在后续实现中保持一致。
- 新增或复用错误码需要同步错误码文档和接口验收。

## US-006 后台运营在列表中识别系统编码

**作为** 后台管理员或内部员工，  
**我希望** 类目列表名称列第一行只看类目名称，第二行只看系统类目编码，  
**以便** 在排查或沟通类目数据时能快速定位编码，同时不被层级路径干扰。

验收要点：

- 名称列第一行只显示类目名称。
- 名称列第二行只显示类目编码。
- 名称列不显示层级路径，例如 `父级类目 / 二级类目`。
- 不改变类目列表列结构、筛选、排序、分页或操作入口。
