# REQ-0067 管理端类目编辑弹窗字段与校验规则

## 1. 需求背景

`REQ-0005-tile-category-management` 已定义管理端瓷砖类目管理能力，包含类目树、列表、创建、更新、启停、删除和编码唯一约束。原始验收中，新增 / 编辑弹窗仍要求展示「类目编码」并由用户填写，类目名称允许 1-30 字符。

产品方现在明确：类目编码是系统内部标识，必须由系统自动生成，不应暴露为运营填写项；类目名称需要更严格的主数据质量约束，最多 10 个字符，仅允许中文、英文和数字，并且同一上级类目下不得重名。该需求用于收敛管理端弹窗和后端兜底规则，避免运营维护成本和重复类目数据进入后续 SKU 挂载、小程序分类展示链路。

本需求为 `REQ-0005-tile-category-management` 的子需求 / refinement，并需要与进行中的 `limit-admin-tile-categories-to-two-levels` 规则保持一致：新增类目最多两级，前端不得允许选择二级类目作为新增类目的上级。

## 2. 目标用户

- **后台管理员 / 内部员工**：在管理端新增或编辑瓷砖类目，维护类目名称、层级、排序和启用状态。
- **后续消费方**：SKU 维护、小程序分类页和商品列表依赖类目主数据，但本需求不直接修改这些前台页面。

## 3. 范围

### 3.1 本期包含

- 管理端新增 / 编辑类目弹窗字段调整。
- 管理端类目列表名称列展示规则调整。
- 类目编码由后端系统自动生成，前端弹窗不展示、不填写、不提交编码输入项。
- 类目列表名称列第一行展示类目名称，第二行仅展示系统类目编码，不展示层级路径。
- 上级类目、类目名称、排序权重必填标识与必填校验。
- 类目名称长度、字符集和同层级唯一校验。
- 前端表单即时校验与后端 API 兜底校验。
- OpenAPI / Orval / API 文档 / 测试按接口契约变化同步。

### 3.2 本期不包含

- 不调整类目列表的列结构、筛选条件、排序策略或分页策略。
- 不实现类目拖拽排序、批量排序或层级迁移。
- 不迁移或清理历史重复类目、历史三级类目或历史不合规名称。
- 不调整小程序分类页 UI、商品列表筛选或 SKU 类目挂载逻辑。
- 不新增多语言类目、类目别名、类目合并、类目导入导出。

## 4. 功能要求

### FR-001 类目编码系统自动生成

- 新增类目时，管理端弹窗 MUST NOT 展示「类目编码」输入项。
- 新增类目时，前端请求 MUST NOT 要求用户填写或提交 `code` 字段。
- 后端 MUST 在创建类目时自动生成唯一类目编码，编码格式 MUST 以 `CAT-` 开头。
- 后端 MUST 继续保证类目编码唯一；发生极端冲突时必须重试生成或返回稳定业务错误，不得把数据库唯一约束原始错误暴露给前端。
- 编辑既有类目时，管理端弹窗 MUST NOT 提供编码修改入口。
- 类目详情、列表或接口响应 MAY 继续包含 `code` 作为内部追踪字段；本需求只禁止在新增 / 编辑弹窗中面向用户展示和填写编码。

### FR-002 上级类目必填表达

- 新增类目弹窗 MUST 展示「上级类目」字段，并带有必填标识。
- 创建一级类目时，用户 MUST 选择固定选项 **「无上级，创建一级类目」** 或等价文案；该选项提交到后端时仍表达为 `parent_id = null`。
- 创建二级类目时，用户 MUST 从可用一级类目中选择上级类目。
- 在最多两级类目规则生效后，前端上级类目下拉 MUST NOT 提供二级类目作为可选上级。
- 编辑既有类目时，本期不允许修改上级类目；如弹窗展示上级类目信息，应为只读展示或不展示，不得允许改变 `parent_id`。

### FR-002A 类目列表名称列展示

- 管理端类目列表名称列 MUST 第一行展示类目名称。
- 管理端类目列表名称列 MUST 第二行展示类目编码。
- 第二行 MUST 仅展示类目编码，不得拼接或展示层级路径，例如 `父级类目 / 二级类目`。
- 本规则仅调整名称列展示内容，不改变类目列表列结构、筛选条件、排序策略、分页策略或 API 响应字段。

### FR-003 类目名称必填与格式校验

- 类目名称字段 MUST 带有必填标识。
- 类目名称 trim 后 MUST 非空。
- 类目名称最多 10 个字符；字符数按用户可见字符计数，中文、英文、数字均按 1 个字符计算。
- 类目名称仅允许中文、英文和数字。
- 类目名称 MUST NOT 接受空格、标点符号、emoji、下划线、连字符、斜杠或其他特殊字符。
- 前端应在保存前给出即时校验提示；后端必须以 API 校验兜底。
- 推荐错误提示：
  - 空名称：`类目名称不能为空`
  - 超长：`类目名称不能超过 10 个字符`
  - 非法字符：`类目名称只能包含中文、英文和数字`

### FR-004 同层级类目名称唯一

- 同一个上级类目下，类目名称 MUST 唯一。
- 创建类目时，如果同一 `parent_id` 下已存在相同名称的类目，后端 MUST 拒绝保存并返回稳定业务错误。
- 编辑类目时，如果名称未变化或只修改排序权重、描述等其他字段，后端 MUST NOT 将当前类目自身误判为重复。
- 编辑类目时，如果把名称改为同一 `parent_id` 下其他类目的名称，后端 MUST 拒绝保存。
- 顶级类目之间的重复校验以 `parent_id = null` 作为同一层级判断。
- 英文大小写是否区分由后续 OpenSpec Change 定稿；默认建议不区分大小写，以降低重复主数据风险。
- 推荐错误提示：`同一层级下已存在同名类目`。

### FR-005 排序权重必填与校验

- 排序权重字段 MUST 带有必填标识。
- 排序权重 MUST 为正整数。
- 前端应在保存前阻止空值、0、负数、小数和非数字输入。
- 后端必须继续用 `CATEGORY_INVALID_SORT_ORDER` 或等价稳定错误码兜底非法排序权重。
- 排序权重的默认值可沿用现有 `10`，但必须允许用户修改为其他正整数。

### FR-006 保存与刷新行为

- 保存成功后，弹窗 MUST 关闭，并刷新类目树、类目列表和页面统计数据。
- 保存失败时，弹窗 MUST 保持打开，并在表单内展示可读错误提示。
- 前端应优先将后端字段级错误映射到对应字段；无法定位字段时展示统一表单错误。
- API 返回结构必须继续遵守统一响应 envelope，不得暴露 FastAPI / Pydantic 默认 `detail` 作为唯一错误结构。

## 5. UI / UE 约束

- 弹窗继续沿用管理端 Design System 与 `CategoryFormModal` 现有 modal 视觉结构。
- 必填标识应出现在字段标签附近，且对「上级类目」「类目名称」「排序权重」三项一致。
- 移除编码输入后，字段顺序建议为：上级类目、类目名称、排序权重、类目描述、状态。
- 不使用解释性大段文案替代表单控件；错误提示应短、明确、可定位。
- 低视口下弹窗 body 需要保持可滚动，底部操作区不得遮挡字段。

## 6. 数据与接口约束

- `tile_categories.code` 字段继续保留，仍为唯一内部编码。
- 创建 API 的请求契约应调整为不要求 `code`；后端负责生成并保存。
- 更新 API 继续不允许修改 `code`。
- 名称校验和同层级唯一校验必须在后端服务层或等价业务层兜底，不能只依赖前端。
- 如新增业务错误码，必须同步 `docs/standards/error-codes.md`、OpenAPI 和测试。
- 若接口请求 / 响应 schema 变化，必须同步 Orval 生成客户端，前端不得手写重复接口类型。
