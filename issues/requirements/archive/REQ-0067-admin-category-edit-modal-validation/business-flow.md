# 业务流程

## 1. 新增类目主流程

```text
后台用户打开类目管理页
  |
  v
点击「新增类目」
  |
  v
系统打开新增类目弹窗
  |-- 上级类目：必填，含「无上级，创建一级类目」和可用一级类目
  |-- 类目名称：必填，中文/英文/数字，最多 10 个字符
  |-- 排序权重：必填，正整数
  |-- 类目描述：可选
  |-- 状态：沿用现有创建状态规则
  |
  v
用户点击「保存类目」
  |
  +-- 前端校验失败 --> 字段下方展示错误，弹窗保持打开
  |
  +-- 前端校验通过
        |
        v
      POST /api/v1/admin/tile-categories
        |
        |-- 请求不提交 code
        |-- 后端校验名称、上级、排序、层级、同层级名称唯一
        |-- 后端生成 CAT- 前缀唯一 code
        |
        +-- 成功 --> 关闭弹窗，刷新类目树/列表/统计
        |
        +-- 失败 --> 返回统一 envelope，弹窗展示可读错误
```

## 2. 编辑类目主流程

```text
后台用户在类目行点击「编辑」
  |
  v
系统打开编辑类目弹窗
  |-- 不展示/不允许编辑类目编码
  |-- 本期不允许修改上级类目
  |-- 允许编辑类目名称、排序权重、类目描述
  |
  v
用户保存
  |
  +-- 名称未变化 --> 不因自身名称误判重复
  |
  +-- 名称变化
        |
        +-- 新名称与同层级其他类目重复 --> 拒绝保存
        |
        +-- 新名称合法且不重复 --> 更新名称与 path，刷新树/列表/统计
```

## 3. 异常与兜底流程

| 场景 | 前端行为 | 后端兜底 | 推荐提示 |
|---|---|---|---|
| 上级类目为空但未选择固定顶级选项 | 阻止保存 | 可接受 `parent_id = null` 表达顶级 | 请选择上级类目 |
| 类目名称为空 | 字段提示 | 拒绝 | 类目名称不能为空 |
| 类目名称超过 10 字符 | 字段提示 | 拒绝 | 类目名称不能超过 10 个字符 |
| 类目名称含空格或符号 | 字段提示 | 拒绝 | 类目名称只能包含中文、英文和数字 |
| 同层级名称重复 | 可选前端预检；保存后处理 API 错误 | 拒绝并返回稳定业务错误 | 同一层级下已存在同名类目 |
| 排序权重非法 | 字段提示 | `CATEGORY_INVALID_SORT_ORDER` 或等价错误 | 排序权重必须为正整数 |
| 编码生成冲突 | 不感知，后端重试 | 重试或返回稳定业务错误 | 保存失败，请重试 |

## 4. 与父需求差异

| 维度 | REQ-0005 基线 | REQ-0067 调整 |
|---|---|---|
| 类目编码 | 新增弹窗展示并要求用户填写；后端校验唯一 | 弹窗不展示、不填写、不提交；后端自动生成 `CAT-` 前缀唯一编码 |
| 列表名称列 | 名称列可混合展示编码与层级路径 | 第一行仅类目名称，第二行仅类目编码，不展示层级路径 |
| 类目名称长度 | 1-30 字符 | 最多 10 个字符 |
| 类目名称字符集 | 未明确限制 | 仅中文、英文、数字 |
| 类目名称重复 | 主要约束编码唯一 | 新增同一上级类目下名称唯一 |
| 上级类目 | 为空时创建一级类目 | 弹窗中以必填固定选项表达顶级语义 |
| 层级上限 | 历史最多三级 | 与两级类目 Change 对齐，不提供二级类目作为新增上级 |

## 5. 系统边界

- Web 管理端：调整 `CategoryFormModal` 字段、必填标识、本地校验和 API payload。
- Backend/API：调整创建 Schema、业务校验、编码生成和错误码。
- Orval：创建请求类型变化后必须重新生成并更新调用方。
- 数据库：保留 `tile_categories.code` 唯一字段；默认不做结构迁移。
- 小程序：不直接改动，只消费更干净的后台类目主数据。
