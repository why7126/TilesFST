# 验收标准

## 1. 弹窗字段与 UI

- [ ] **AC-001** 新增类目弹窗不展示「类目编码」输入项，也不展示要求用户填写编码的帮助文案。
- [ ] **AC-002** 编辑类目弹窗不展示可编辑的「类目编码」输入项，不允许通过弹窗修改既有编码。
- [ ] **AC-003** 新增 / 编辑弹窗字段顺序为：上级类目（新增时）、类目名称、排序权重、类目描述、状态（如创建态沿用现有规则）。
- [ ] **AC-004** 上级类目、类目名称、排序权重标签附近均有一致的必填标识。
- [ ] **AC-005** 新增一级类目时，上级类目通过「无上级，创建一级类目」或等价固定选项表达，提交语义为 `parent_id = null`。
- [ ] **AC-006** 新增二级类目时，上级类目只能选择可用一级类目；二级类目不得出现在上级类目可选项中。
- [ ] **AC-007** 编辑既有类目时，本期不允许修改上级类目。
- [ ] **AC-008** 字段级校验错误展示在对应字段或字段组下方，无法映射字段的服务端错误展示在固定表单错误区。

## 1A. 类目列表名称列

- [ ] **AC-008A** 管理端类目列表名称列第一行只显示类目名称。
- [ ] **AC-008B** 管理端类目列表名称列第二行只显示类目编码。
- [ ] **AC-008C** 管理端类目列表名称列第二行不显示层级路径，例如 `父级类目 / 二级类目`。
- [ ] **AC-008D** 类目列表列结构、筛选条件、分页与操作列不因本展示规则变更而回归。

## 2. 类目名称校验

- [ ] **AC-009** 类目名称 trim 后为空时，前端阻止保存并提示「类目名称不能为空」或等价文案。
- [ ] **AC-010** 类目名称超过 10 个用户可见字符时，前端阻止保存并提示「类目名称不能超过 10 个字符」或等价文案。
- [ ] **AC-011** 类目名称包含空格、标点、emoji、下划线、连字符、斜杠或其他特殊字符时，前端阻止保存并提示「类目名称只能包含中文、英文和数字」或等价文案。
- [ ] **AC-012** 类目名称允许中文、英文、数字及中英文数字混排，且中文、英文、数字均按 1 个字符计数。
- [ ] **AC-013** 后端 API 对空名称、超过 10 字符名称和非法字符名称均兜底拒绝，返回统一 envelope。

## 3. 同层级名称唯一

- [ ] **AC-014** 创建顶级类目时，如果已存在同名顶级类目，后端拒绝保存并返回稳定业务错误。
- [ ] **AC-015** 创建二级类目时，如果同一一级父类目下已存在同名二级类目，后端拒绝保存并返回稳定业务错误。
- [ ] **AC-016** 不同上级类目下允许存在同名二级类目，除非后续 OpenSpec 明确收紧为全局唯一。
- [ ] **AC-017** 编辑类目且名称未变化时，不因自身记录误判为重复。
- [ ] **AC-018** 编辑类目并改名为同层级其他类目名称时，后端拒绝保存。
- [ ] **AC-019** 英文大小写重复规则在 OpenSpec Change 中定稿；若采用不区分大小写，则 `ABC` 与 `abc` 在同层级视为重复。

## 4. 编码生成与接口契约

- [ ] **AC-020** `POST /api/v1/admin/tile-categories` 请求不再要求前端提交 `code` 字段。
- [ ] **AC-021** 后端创建类目时自动生成唯一编码，返回对象中的 `code` 以 `CAT-` 开头。
- [ ] **AC-022** 后端保留 `tile_categories.code` 唯一约束，编码冲突时不得暴露数据库原始异常。
- [ ] **AC-023** `PUT /api/v1/admin/tile-categories/{id}` 继续不允许修改 `code` 或 `parent_id`。
- [ ] **AC-024** API 错误响应继续遵守 `{ code, message, data }` envelope；Pydantic 校验错误不得只暴露 FastAPI 默认 `detail`。
- [ ] **AC-025** 如新增类目名称重复错误码，必须登记到 `docs/standards/error-codes.md` 并在测试中断言。
- [ ] **AC-026** OpenAPI 导出和 Orval 生成物与创建请求契约保持一致，前端不手写重复接口类型。

## 5. 排序权重与保存刷新

- [ ] **AC-027** 排序权重为空、0、负数、小数或非数字时，前端阻止保存。
- [ ] **AC-028** 后端继续拒绝非法排序权重，并返回 `CATEGORY_INVALID_SORT_ORDER` 或等价稳定错误。
- [ ] **AC-029** 保存成功后弹窗关闭，并刷新类目树、类目列表和页面统计数据。
- [ ] **AC-030** 保存失败后弹窗保持打开，已填写字段不丢失。

## 6. 测试与回归

- [ ] **AC-031** Web 组件测试覆盖：编码输入不出现、必填标识出现、上级类目只含顶级选项、名称格式校验、排序权重校验。
- [ ] **AC-032** 后端测试覆盖：创建不传 `code` 成功且生成 `CAT-` 编码。
- [ ] **AC-033** 后端测试覆盖：非法名称、超长名称、同层级重复名称、编辑自身不误判、编辑改为重复名失败。
- [ ] **AC-034** API / Orval 变更后，前端调用 payload 不再包含用户填写的 `code`。
- [ ] **AC-035** 回归类目列表、类目树、启停、删除、SKU 挂载的既有能力不受影响。

## 横切 AC（knowledge-base）

> 来源：`docs/knowledge-base/best-practices/admin-modal-width-css-cascade.md` — 预防 Sprint 003 弹窗宽度与 CSS cascade 复发类缺陷。

- [ ] **AC-XCUT-001** `CategoryFormModal` 或后续专属弹窗 TSX 不得同时挂载通用 `modal-card` 与专属类；应使用单一专属类（如 `category-modal` / `category-modal-card`）承载宽度样式。
- [ ] **AC-XCUT-002** 在 1440px 视口下验收弹窗 computed width 与设计批准宽度一致；本 REQ 默认沿用类目弹窗 560px，若 Change 另行调整宽度，必须在 design/tasks 中写明。
- [ ] **AC-XCUT-003** Web 测试或等价 CSS 栈校验需覆盖可能冲突的 admin CSS，确认类目弹窗宽度未被 `.modal-card`、用户管理或系统设置样式覆盖。
- [ ] **AC-XCUT-004** 矮视口下弹窗 body 可滚动，底部操作区不遮挡上级类目、类目名称、排序权重、描述或状态字段。
