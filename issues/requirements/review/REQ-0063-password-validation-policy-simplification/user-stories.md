# 用户故事

## US-001 管理端用户按新规则修改本人密码

作为管理端用户，我希望修改密码时看到清晰的新密码规则，以便用 5-32 位且包含英文和数字的密码完成修改。

### 验收要点

- 新密码为 5-32 位、包含英文字符、包含数字时可以进入提交。
- 新密码少于 5 位、超过 32 位、缺少英文字符或缺少数字时，表单在对应字段附近提示原因。
- 页面或弹窗不得继续展示“至少 8 位”等旧规则。
- 修改本人密码仍需校验原密码，且不改变成功后的登出或 token 失效策略。

## US-002 管理员创建或重置用户密码

作为管理端管理员，我希望创建用户或重置用户密码时使用同一套密码规则，以便不同入口不会出现规则漂移。

### 验收要点

- 创建用户入口如支持输入初始密码，使用 5-32 位、英文、数字规则。
- 重置用户密码入口如支持输入或生成新密码，输入值或生成值均满足新规则。
- 后端返回的策略错误 message 与管理端字段提示一致。
- 所有相关入口复用统一校验逻辑或等价公共函数。

## US-003 系统维护人员验证密码边界

作为系统维护人员，我希望通过测试覆盖密码规则边界，以便后续调整不会回退到旧的 8 位策略或多入口分叉。

### 验收要点

- 测试覆盖 4 位失败、5 位成功、32 位成功、33 位失败。
- 测试覆盖纯英文失败、纯数字失败、英文加数字成功。
- 测试覆盖包含符号且满足英文/数字/长度时成功，除非后续 OpenSpec 明确收紧允许字符集。
- 测试覆盖修改密码、创建用户、重置密码中至少一个后端集成入口，并覆盖公共校验函数。

## US-004 产品验收密码提示文案

作为产品验收人员，我希望所有密码输入位置展示一致、可理解的新规则文案，以便用户不会因旧提示或模糊提示误判。

### 验收要点

- 通用规则文案为“密码需为 5-32 位，并同时包含英文字符和数字。”或等价表达。
- 字段级错误能区分长度不足、长度超限、缺英文、缺数字。
- 服务端无法映射到具体字段的错误可放在固定表单错误区，但不得只用全局 Toast 承载字段级密码错误。
