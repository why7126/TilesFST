# 临时规避方案

## 1. 用户侧规避

在正式修复前，用户可按当前默认安全策略主动构造更强密码，降低反复试错概率：

1. 至少 12 位字符。
2. 同时包含大写字母、小写字母、数字和特殊字符。
3. 不与原密码相同。
4. 不使用常见弱密码或公司/产品名称加简单数字的组合。

示例格式：

```text
Tile2026#Secure
Stone@2026Admin
```

以上仅为格式示例，不应作为真实密码直接使用。

## 2. 管理侧规避

管理员或运维人员可在用户遇到该问题时，临时告知当前有效密码策略，尤其是最小长度和是否必须包含特殊字符。

如系统设置中调整过密码策略，应以“系统设置 > 安全策略”中的当前配置为准，避免继续引用前端弹窗里的静态旧规则。

## 3. 不可规避项

以下问题无法通过操作规避彻底解决：

- 后端返回的策略失败原因仍是粗粒度错误。
- 前端规则列表不会自动反映当前有效策略。
- 用户仍可能不知道具体缺少哪一类字符。
- 若后续策略继续调整，前端静态提示仍可能再次失准。

## 4. 风险说明

临时规避不会影响认证安全边界，但存在可用性风险：

- 用户多次尝试可能触发改密频率限制。
- 用户可能选择过度复杂但难记的密码，增加后续登录失败概率。
- 管理员口头说明策略容易与真实配置漂移。
