# 临时规避方案

## 1. 可用规避

在微信开发者工具或真机 evidence 补齐前，不应把 Sprint 008 小程序首页表述为“真机验收已通过”。

如需要临时汇报或演示，可采用以下方式降低误判：

1. 明确标注当前结论为“自动化与静态测试通过，DevTools / 真机验收待补齐”。
2. 展示 `tests/test_miniapp_static.py` 的通过结果作为代码形态侧证。
3. 展示 Sprint 008 验收报告中保留的人工 follow-up 风险，避免把强制关闭误解为完整设备验收通过。
4. 演示小程序首页时同步记录当前设备、逻辑宽度和截图，不把未记录设备信息的截图作为正式 evidence。

## 2. 验收前检查清单

人工复验前建议先确认：

- 微信开发者工具打开的是 `src/miniapp/`。
- `pages/index/index` 能进入首页。
- 本地接口地址可访问，必要时确认开发者工具本地 URL 校验设置。
- 首页存在可展示的 Banner 或公开商品数据；若无数据，需要区分“数据为空降级”和“布局遮挡”。
- 预览宽度覆盖 320、375、390、430 pt。

## 3. 无产品内规避

当前没有产品内开关、后台配置或环境变量可以替代设备验收。静态测试通过、OpenSpec 已归档、Sprint 已关闭都不能自动推导出真机视觉通过。

## 4. 风险说明

| 风险 | 说明 |
|---|---|
| 验收误判 | 高；如果没有截图 evidence，容易把自动化通过误写为真机通过 |
| 用户体验 | 高；若真实设备存在遮挡，终端客户首页首屏会直接受影响 |
| 发布表述 | 中；发布说明必须保留设备验收边界，避免过度承诺 |
| 修复范围扩大 | 中；若真机发现遮挡，后续可能需要小程序 UI fix Change |

## 5. 规避有效期

- **有效期**：自 `BUG-0068` capture 起，至 DevTools / 真机 evidence 补齐并通过评审。
- **闭环后**：以 `acceptance.md` 中要求的截图、设备信息和复验结论为准。
