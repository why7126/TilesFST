# 根因分析

## 1. 分类

| 项 | 值 |
|---|---|
| 缺陷类型 | **code / build / miniapp-runtime** |
| 引入阶段 | `add-miniapp-home` 小程序首页实现阶段 |
| 责任模块 | 微信小程序首页 `src/miniapp/pages/index/` 与小程序运行入口 |
| 关联后端 | 后端首页聚合接口可作为数据来源，但当前首要问题是小程序运行时代码未接上 |
| 关联测试 | 小程序静态测试只检查 `.ts` 源码字符串，未校验微信开发者工具实际加载的 `.js` 页面脚本 |

## 2. 直接原因

微信小程序首页目录同时存在 `index.ts` 与 `index.js`：

- `src/miniapp/pages/index/index.ts` 包含首页 `Page` 数据、`loadHome()`、`/api/v1/miniapp/home` 请求和页面交互逻辑。
- `src/miniapp/pages/index/index.js` 仍是微信开发者工具生成的空模板，`data` 为空，`onLoad` 不调用 `loadHome()`。

当前微信开发者工具预览表现与空 `index.js` 被加载一致：WXML 中的静态文案可以显示，但依赖 `home` 数据的 Banner、快捷入口、新品推荐、热销推荐和服务卡片没有正常渲染。

## 3. 根本原因

### 3.1 小程序源码事实源不一致

实现阶段把业务逻辑写入 `.ts` 文件，但当前小程序工程没有明确、可验证的 TypeScript 编译链把 `.ts` 同步为运行时 `.js`。同目录空 `.js` 保留后，开发者工具优先执行运行时 `.js`，导致真实预览未运行已实现逻辑。

### 3.2 小程序构建/预览门禁缺失

`project.config.json` 未体现可依赖的 TS 编译插件或构建输出策略，项目也缺少“页面 `.ts` 与 `.js` 不得脱节”的校验。实现完成时没有阻止空模板 `.js` 与业务 `.ts` 并存。

### 3.3 测试覆盖了源码意图，未覆盖运行入口

现有小程序静态测试检查 `pages/index/index.ts` 中是否包含接口路径和埋点字符串，因此无法发现 `pages/index/index.js` 仍为空模板。后端聚合接口测试也只能证明 API 有数据时可用，不能证明微信预览实际调用了该接口。

## 4. 触发条件

满足以下条件即可稳定复现：

1. 使用微信开发者工具打开 `src/miniapp/`。
2. 首页目录保留空模板 `pages/index/index.js`。
3. 小程序预览执行 `.js` 运行入口，而不是直接执行 `.ts` 源码。
4. 进入首页 `pages/index/index`。

触发后，页面静态结构仍可显示，但动态首页数据不会加载，首屏呈现空商品提示和缺失模块。

## 5. 次级触发因素

| 因素 | 说明 |
|---|---|
| 本地后端未启动或接口地址不可访问 | 会导致首页数据加载失败，但当前截图未出现网络错误态，且首要证据指向运行入口未执行 |
| 本地缺少公开商品或 Banner 数据 | 会影响推荐模块和 Banner 展示，但不应导致快捷入口、服务卡片等由前端/默认服务生成的模块全部消失 |
| 小程序请求域名配置 | 开发者工具预览本地接口时需要确认 `localhost:8000` 可访问；这是修复验收时必须验证的环境条件 |

## 6. 非根因（排除）

| 假设 | 结论 |
|---|---|
| 后端首页聚合接口完全不可用 | 暂不作为首要根因；相关测试显示有数据时接口能返回公开 Banner、商品和服务区字段 |
| 只有商品数据缺失 | 否；快捷入口和服务区也缺失，说明不只是商品列表为空 |
| 原型本身不可实现 | 否；当前代码已包含接近原型的 WXML/WXSS 与 TS 逻辑，问题在运行入口/构建链 |
| 底部 TabBar 配置错误 | 否；截图中 TabBar 可见，主要问题不在路由注册 |
| 数据库结构缺失 | 暂无证据；本 BUG 修复默认不需要 DB schema 变更 |

## 7. 分类结论

| 维度 | 结论 |
|---|---|
| 缺陷分类 | code / build / miniapp-runtime |
| 是否接口缺陷 | 默认否；修复时需复验接口可访问 |
| 是否数据库缺陷 | 否 |
| 是否权限缺陷 | 否 |
| 是否测试缺口 | 是；缺少实际运行入口与预览 smoke 校验 |
| 主要修复面 | 小程序运行入口、构建链或 `.js` 同步策略，以及相关静态/预览校验 |

## 8. 修复建议（供 bug-opsx）

1. 明确 `src/miniapp/` 的运行事实源：要么使用可验证的 TypeScript 编译链生成 `.js`，要么将业务逻辑落到微信开发者工具实际加载的 `.js`。
2. 清理或同步空模板 `.js`，确保首页、搜索、详情、门店页等页面运行脚本不再与 `.ts` 脱节。
3. 增加小程序静态校验：当同一页面同时存在 `.ts` 与 `.js` 时，禁止 `.js` 保持空模板或缺少关键 Page 数据/方法。
4. 修复后在微信开发者工具中复验首页首屏，并确认 `/api/v1/miniapp/home` 被实际请求。
5. 默认不修改后端 API、数据库、Orval 或 Docker Compose；如修复中发现接口契约或环境变量必须调整，需要在对应 OpenSpec Change 中说明。
