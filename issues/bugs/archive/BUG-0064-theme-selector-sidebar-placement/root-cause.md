# 根因分析

## 1. 分类

| 项 | 值 |
|---|---|
| 缺陷类型 | **design / frontend-ui / layout** |
| 引入阶段 | 主题偏好入口接入管理端 Shell 时 |
| 责任模块 | Web 管理端 Shell / 侧边栏布局 |
| 关联后端 | 无；默认不涉及 API、数据库、鉴权或 Orval |

## 2. 直接原因

管理端「界面主题」选择器已经作为全局偏好入口接入，但当前放置在页面右上角内容区，而不是侧边栏底部用户信息块上方。

该放置方式让一个全局 Shell 偏好控件占用了页面级顶部操作区域，并与侧边栏中的用户身份信息分离，导致用户期望的“主题入口贴近个人/偏好区域”的信息层级未被满足。

## 3. 根本原因

### 3.1 全局偏好控件缺少侧边栏归属约束

主题切换属于全局偏好，不属于单个业务页面动作。现有实现只完成了控件展示与选择能力，但没有明确约束它在管理端 Shell 中应归属于侧边栏用户区附近。

### 3.2 主题入口验收未覆盖布局位置

关联需求 `REQ-0020-theme-comfort-refine` 关注主题舒适度与主题选择能力，但当前缺陷暴露出验收粒度没有覆盖「主题选择器必须位于用户头像上方」这一具体布局要求。

### 3.3 页面顶部操作区与 Shell 偏好区边界不清

右上角区域通常承载当前页面动作、筛选或跳转入口。主题选择器是跨页面偏好入口，放在该区域会造成职责混杂；应由侧边栏承载全局导航、用户身份与偏好相关入口。

## 4. 触发条件

满足以下条件即可稳定复现：

1. 管理员登录 Web 管理端；
2. 进入任意管理端 Shell 页面，例如「接口文档」；
3. 页面右上角显示「界面主题 / 系统默认」选择器；
4. 左侧侧边栏底部用户头像/账号信息块上方没有主题选择入口。

## 5. 非根因（排除）

| 假设 | 结论 |
|---|---|
| 主题选择能力不可用 | 否；问题是展示位置不符合期望 |
| 后端接口错误 | 否；这是前端 Shell 布局问题 |
| 数据库结构缺失 | 否；不涉及新增或修改表结构 |
| 用户权限导致不可见 | 否；截图显示控件可见，但位于错误区域 |
| Design Token 缺失 | 否；主要问题是组件放置位置与信息层级 |

## 6. 分类结论

| 维度 | 结论 |
|---|---|
| 缺陷分类 | design / frontend-ui / layout |
| 是否接口缺陷 | 否 |
| 是否数据库缺陷 | 否 |
| 是否权限缺陷 | 否 |
| 是否 Design System 缺陷 | 部分是；Shell 布局与全局偏好入口位置需对齐 |
| 主要修复面 | 管理端 Shell 布局与主题选择器放置位置 |

## 7. 修复建议（供 bug-opsx）

1. 将主题选择器从页面右上角内容区移入左侧侧边栏。
2. 主题选择器位置应位于侧边栏底部用户头像/账号信息块上方。
3. 保持主题选择器原有选择能力与状态持久化行为不变。
4. 适配侧边栏展开/收起状态，避免控件文本、图标、下拉框与用户信息区重叠。
5. 默认不修改后端 API、数据库、Orval 生成物；如实现时发现已有主题偏好接口需要调整，必须在 OpenSpec Change 中说明。
