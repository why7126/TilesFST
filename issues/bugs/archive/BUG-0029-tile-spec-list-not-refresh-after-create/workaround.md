# 临时规避方案

## 1. 运营规避（当前唯一有效手段）

保存规格后若列表未出现新记录或变更未反映：

1. **手动刷新浏览器**（F5 或 Cmd+R）——数据已在后端，刷新后必可见。
2. 以 Toast「规格已创建」/「规格已更新」为准判断保存是否成功，勿重复提交相同宽长组合（会触发服务端唯一性错误）。
3. 连续新增多条时，每条保存后执行一次 F5，或保存后点击「查询」按钮（会触发 `loadSpecs(1)`，等效刷新当前筛选下的列表）。

## 2. 验收规避

在正式修复前，验收 REQ-0009 规格 CRUD 时应：

- 将「保存后列表即时更新」**标注为已知缺陷 BUG-0029**，暂不作为通过项。
- 启停、删除、筛选、分页刷新行为可单独验证（这些路径正常）。
- 后端 API 与 DB 持久化可通过 Network 面板或 API 直接查询验证，不依赖列表 UI。

## 3. 开发规避

若需在未修复分支上临时验证数据：

- 保存后调用浏览器 DevTools → Network，确认 `POST /api/v1/admin/tile-specs` 返回 200。
- 或保存后点击筛选区「查询」按钮强制 `loadSpecs`。

## 4. 风险说明

| 风险 | 说明 |
|---|---|
| 重复提交 | 用户误以为保存失败，可能重复点击保存，触发 `TILE_SPEC_DUPLICATED` |
| 效率损失 | 每条维护操作需额外 F5，批量建档体验差 |
| 误删/误操作 | 在 stale 列表上可能对错误行执行启停/删除（数据本身仍正确，但认知负担高） |

**结论：** 无代码层零成本规避；须通过 `/bug-review` → `fix-*` Change 修复 `onSuccess` 回调。
