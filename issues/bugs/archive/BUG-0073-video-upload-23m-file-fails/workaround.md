# Workaround - BUG-0073 上传 23M 文件失败，图片、视频和文档均受影响

## 临时规避方案

在正式修复前，可采用以下方式降低影响：

1. 图片文件：临时压缩到当前有效 `MAX_IMAGE_SIZE_MB` / `media.max_image_size_mb` 以下，默认建议小于 20MB。
2. 文档 / 证书文件：临时压缩 PDF 或拆分附件，使单个文件小于 20MB。
3. 视频文件：确认使用视频上传入口，并使用允许的视频 MIME Type；若 23M 视频仍失败，记录 HTTP 状态码、响应 JSON、请求 URL 和后端日志，优先排查代理限制或对象存储写入失败。
4. 管理端操作：上传前先确认页面提示的支持格式和大小限制；如页面未提示，以后端当前限制为准。
5. 运营处理：对必须上传的大文件，先保留原始文件和失败证据，等待修复后统一补录。

## 不建议的规避方式

- 不建议关闭后端大小校验、MIME Type 校验或鉴权来临时放行上传。
- 不建议让前端直接写入 MinIO / S3 兼容对象存储。
- 不建议把真实客户文件、生产附件或大文件样本提交到 Git 仓库。
- 不建议通过修改对象 Key 或使用原始文件名绕过上传流程。
- 不建议仅修改前端提示而不修正后端、代理和系统设置的一致性。

## 运维 / 客服说明

如用户反馈约 23M 文件上传失败，可先说明：

1. 当前系统对图片、视频、文档存在不同上传大小限制。
2. 图片和证书 / PDF 类文件默认可能被 20MB 限制拦截。
3. 视频 23M 理论上应在默认 500MB 限制内，需要根据具体入口和部署环境进一步排查。
4. 正式修复会统一文件类型、大小限制、错误提示和回归测试。

## 风险

临时压缩文件可能降低图片或视频质量，拆分文档会增加资料维护成本。任何绕过后端上传校验的方案都会带来安全、权限和对象存储一致性风险，不应采用。
