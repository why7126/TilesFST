# Root Cause - BUG-0073 上传 23M 文件失败，图片、视频和文档均受影响

## 直接原因

公共上传能力对不同文件类型的大小限制不一致，且部分限制与用户对“约 23M 文件可上传”的业务预期不一致：

- 图片上传入口使用 `media.max_image_size_mb` / `MAX_IMAGE_SIZE_MB`，默认上限为 20MB。约 23M 图片会被 `save_upload_file()` 判定为 `FILE_SIZE_EXCEEDED`。
- 品牌证书 / PDF 文件上传入口使用代码内常量 `CERTIFICATE_MAX_SIZE_MB = 20`，未纳入 `EffectiveSettingsService` 的媒体配置体系。约 23M PDF 或证书类文件会被拒绝，并返回“证书文件不能超过 20MB”。
- 视频上传入口使用 `media.max_video_size_mb` / `MAX_VIDEO_SIZE_MB`，默认上限为 500MB；若约 23M 视频仍失败，需要进一步确认是否使用了非视频上传入口、生产代理层请求体限制、对象存储写入失败、请求超时或 MIME Type 与入口不匹配。

## 根本原因

根本原因是上传规格缺少跨类型的一致治理：

- 图片、视频、文件/证书三类上传没有统一的“业务可接受大小”说明和一致配置入口。
- 图片和文件/证书限制默认 20MB，而视频默认 500MB，用户在上传约 23M 文件时会遇到不同类型表现不一致。
- 文件/证书类上传使用硬编码上限，不能通过系统设置或环境变量与图片、视频统一调整。
- 错误提示缺少跨入口一致的诊断信息，用户很难区分是大小限制、类型限制、入口错误、代理限制还是对象存储失败。
- 当前测试主要覆盖“超限会被拒绝”，缺少“约 23M 的图片/视频/文档边界文件在目标业务规则下应成功或应给出清晰拒绝原因”的回归场景。

## 触发条件

满足以下条件时可触发或疑似触发：

1. 上传文件大小约为 23MB。
2. 文件类型为图片，且当前有效配置 `media.max_image_size_mb` 为 20MB 或低于 23MB。
3. 文件类型为 PDF / 证书类文档，且走品牌证书上传入口。
4. 文件类型为视频但未走 `/api/v1/admin/uploads/tile-videos`，或部署链路代理/网关/对象存储配置低于 23MB。
5. 前端没有在上传前展示与后端一致的大小限制，导致用户只能在提交后看到失败。

## 分类

| 分类 | 判断 |
|---|---|
| config | 是。图片与视频上限来自可配置项，默认值差异直接影响 23M 文件表现 |
| design | 是。图片、视频、文档上传限制缺少统一产品规格和提示策略 |
| code | 是。证书文件上传存在硬编码 20MB 上限，且未纳入统一配置 |
| api | 可能。若后续统一错误响应或暴露上传限制查询接口，需要同步 OpenAPI / Orval |
| db | 可能。若后续把文件类大小限制加入系统设置，需要同步设置项初始化和测试 |
| storage | 可能。视频 23M 失败仍需验证是否存在 MinIO / S3 写入或生产代理配置问题 |
| security | 是。修复必须保留大小、MIME Type、扩展名、鉴权和对象 Key 安全校验 |

## 影响判断

该缺陷影响媒体资料维护的核心链路。图片和文档类约 23M 文件在默认 20MB 限制下会稳定失败；视频类 23M 文件按现有默认配置理论上应成功，因此视频失败更可能暴露入口误用、生产配置漂移或存储/代理链路问题。后续修复应先明确产品允许的图片、视频、文档大小上限，再统一前端提示、后端校验、系统设置、代理配置和回归测试。
