# 临时规避方案

## 1. 操作规避

当前没有稳定的产品内规避方案可以让业务上传对象自动进入 MinIO。上传链路在代码层面写入本地 `UPLOAD_DIR`，因此仅通过重启 MinIO、重新创建桶或刷新 MinIO Console 不能解决问题。

在正式修复前，内部开发和演示可临时采用以下方式降低影响：

1. 若只验证前端上传与页面回显，可临时查看 `/media/{object_key}` 或本地 `data/uploads/` 下的文件。
2. 若验证对象存储集成能力，必须标记该项未通过，不应把 `data/uploads/` 中存在文件视为 MinIO 验收通过。
3. 演示时避免承诺“上传后已进入 MinIO 对象存储”，除非后续修复已完成并回归通过。
4. 对需要长期保留的真实业务素材，不建议依赖当前本地上传目录作为正式存储。

## 2. 验收规避

在正式修复前，媒体上传验收应拆分：

- 上传接口是否返回成功：可单独记录。
- 前端是否能预览或回显：可单独记录。
- MinIO 桶内是否存在业务对象：必须标记为本 BUG 未通过。
- Docker Compose 中 MinIO 是否启动并创建桶：可单独记录，但不能代表业务上传链路已接入对象存储。

## 3. 风险说明

该规避不能消除以下风险：

- Docker 环境中的业务媒体对象和 MinIO bucket 状态不一致。
- 本地 `data/uploads/` 被清理、未挂载或迁移时，已上传素材可能无法访问。
- 小程序或外部访问链路若依赖对象存储 URL，将无法验证真实可用性。
- 后续切换 MinIO 后，已有本地文件可能需要迁移或重新上传。

因此建议进入 `/bug-review`，评审通过后通过 `fix-*` OpenSpec Change 修复。
