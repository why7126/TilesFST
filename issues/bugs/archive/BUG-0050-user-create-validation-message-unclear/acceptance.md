# 验收标准

## AC-001 用户名长度不足返回明确中文提示

- [ ] 使用管理员 token 调用 `POST /api/v1/admin/users`。
- [ ] 请求体 `username` 为 `abc`，`role` 为任一合法角色。
- [ ] API 不返回 FastAPI 默认 422 `detail` 结构。
- [ ] API 返回项目统一错误结构 `{ code, message, data }`。
- [ ] `message` 明确包含用户名长度问题，例如「用户名长度须为 4–32 位」。

## AC-002 前端创建用户弹窗展示具体错误

- [ ] 管理端打开 `/admin/users`。
- [ ] 点击「添加用户」。
- [ ] 输入 `abc` 并提交。
- [ ] 弹窗内或等价错误提示区域显示具体错误原因。
- [ ] 用户能判断需要修改「用户名」字段。

## AC-003 用户名其他格式校验仍明确

- [ ] 输入 `1abc` 时，提示用户名须以小写字母开头或等价明确文案。
- [ ] 输入 `ab__cd` 时，提示用户名不允许连续特殊符号或等价明确文案。
- [ ] 输入保留字时，提示用户名为系统保留字或等价明确文案。

## AC-004 重复用户名错误不回退

- [ ] 创建已存在用户名。
- [ ] API 返回 HTTP 409 或既有等价冲突状态。
- [ ] 错误码与文案仍明确表示用户名已存在。
- [ ] 前端展示「用户名已存在」或等价文案。

## AC-005 合法创建流程不受影响

- [ ] 输入合法用户名、合法角色和可选昵称/头像后可创建成功。
- [ ] 创建成功后仍 Toast「用户已创建」。
- [ ] 如 API 返回 `initial_password`，一次性密码弹窗仍正常展示。

## AC-006 回归测试覆盖

- [ ] 后端测试覆盖 `username="abc"` 时统一错误结构与中文 `message`。
- [ ] 后端测试覆盖至少一个业务层格式错误路径，例如 `username="1abc"`。
- [ ] 前端组件测试覆盖创建失败时展示后端 `message`。
- [ ] 若修复涉及全局 422 错误处理，补充至少一个非用户管理接口或通用请求体验回归测试。
