# BUG-0050 创建用户校验失败未明确提示具体问题点

## 现象

管理端用户管理页创建用户时，如果用户名不满足既有规范，界面未明确提示具体字段和不符合项。典型场景是输入小于 4 位的用户名后提交，用户只能看到泛化失败提示，无法判断应修改用户名长度、格式还是其他字段。

该问题属于 `REQ-0005-user-management` 已交付能力中的反馈缺陷。需求验收已要求用户名必填、4-32 位、仅限小写字母/数字/`_`/`-`/`.`、须以字母开头、禁止连续特殊符号与保留关键字。

## 复现步骤

1. 使用后台管理员账号进入管理端。
2. 打开 `/admin/users` 用户管理页。
3. 点击「添加用户」打开创建用户弹窗。
4. 在「用户名」输入框中输入 `abc`。
5. 选择任意合法角色并提交创建。
6. 观察弹窗或 Toast 中展示的错误提示。

## 期望结果

- 系统明确提示字段与原因，例如「用户名长度须为 4-32 位」。
- 错误提示应帮助用户定位需要修改的字段，至少覆盖用户名长度不满足规范的场景。
- API 错误响应应符合统一结构 `{ code, message, data }`，前端可直接展示 `message`。

## 实际结果

- 创建失败后界面提示不明确，未指出「用户名」字段和「长度不足」原因。
- 已观察到当用户名为 `abc` 时，后端返回 FastAPI 默认校验错误结构，HTTP 状态为 422，响应体为 `detail` 列表，而不是项目统一错误结构。
- 前端错误提取逻辑主要读取 `response.data.message`；当响应体只有 `detail` 时，会显示兜底失败文案。

## 影响范围

| 范围 | 影响 |
|---|---|
| Web 管理端 | 用户管理「添加用户」弹窗的错误提示体验受影响 |
| 后端 API | `POST /api/v1/admin/users` 在用户名长度不足时返回结构与项目统一错误响应不一致 |
| 用户管理流程 | 管理员无法快速判断创建失败原因，可能重复试错 |
| 关联需求 | `REQ-0005-user-management` 的用户名规范反馈体验不完整 |

## 严重等级说明

严重等级为 `medium`。

理由：

- 该问题不阻断合法用户创建，不影响已有用户登录、列表、编辑、冻结或删除能力。
- 该问题发生在常见表单校验路径，会直接影响后台管理员创建用户时的可恢复性与操作效率。
- 该问题涉及 API 错误结构与前端错误展示的一致性，若同类 Pydantic 校验路径存在于其他表单，可能带来相似体验问题。

## 待确认

- 是否要求所有用户创建字段均展示字段级错误，而不仅是用户名长度。
- 修复策略是前端提交前即时校验、后端统一转换 422 响应，还是两者同时补齐。
- 错误文案是否需要补充到 API 错误码治理文档。
