# 缺陷说明

## 现象

Web 管理端多个列表型页面在页面模块顺序、筛选/搜索区交互、表格末列固定浮动和分页页码呈现上不统一，导致管理员在不同模块间切换时操作习惯和视觉预期不一致。

受影响页面包括：

- 瓷砖 SKU
- 瓷砖品牌
- 瓷砖类目
- 瓷砖规格
- Banner 管理
- 用户管理
- 日志审计
- 接口文档

用户明确期望这些页面按统一的管理端列表页规范整理：

1. 模块顺序固定为「标题模块 → 指标卡模块 → 筛选/搜索模块 → 列表模块」。
2. 筛选/搜索模块的交互和样式以瓷砖 SKU 页为准。
3. 所有页面均不需要【查询】功能，多余查询按钮应移除。
4. 重置按钮尺寸保持一致。
5. 列表模块最后一列统一固定浮动，以接口文档页为准。
6. 分页最多展示 5 个可点击页码；当前只有一个页码时也应保持统一分页控件形态。

## 复现步骤

1. 使用管理员账号登录 Web 管理端。
2. 依次打开「瓷砖 SKU」「瓷砖品牌」「瓷砖类目」「瓷砖规格」「Banner 管理」「用户管理」「日志审计」「接口文档」页面。
3. 对比各页面顶部标题区、指标卡区、筛选/搜索区和列表区的呈现顺序。
4. 查看筛选/搜索区是否存在【查询】按钮，以及重置按钮尺寸是否与瓷砖 SKU 页一致。
5. 查看表格最后一列操作区，在横向滚动或表格宽度不足时确认是否按接口文档页一样固定浮动。
6. 查看底部分页控件，确认可点击页码数量是否最多为 5 个，以及当前只有 1 页时的控件形态是否一致。

## 期望结果

- 上述管理端列表页均按「标题模块 → 指标卡模块 → 筛选/搜索模块 → 列表模块」顺序展示。
- 筛选/搜索模块统一以瓷砖 SKU 页为交互和视觉基线。
- 筛选条件变更后应自动驱动列表筛选或采用既有即时筛选逻辑，不再展示多余【查询】按钮。
- 重置按钮在各页面尺寸、边框、圆角、字号和对齐方式一致。
- 列表最后一列操作区统一采用固定浮动样式，视觉和交互以接口文档页为基线。
- 分页控件最多展示 5 个可点击页码；当数据只有 1 页时，分页控件仍保持统一结构和禁用态表达。

## 实际结果

- 部分页面模块顺序与目标顺序不一致，指标卡、筛选/搜索区和列表模块位置存在差异。
- 部分页面仍展示【查询】按钮，和瓷砖 SKU 页筛选/搜索交互不一致。
- 不同页面的重置按钮尺寸和视觉密度不完全一致。
- 表格最后一列在多个页面未统一固定浮动，横向滚动或宽表格场景下操作列可达性不一致。
- 分页页码呈现不统一，当前仅一个页码时也未形成统一可复用的分页行为和视觉基线。

## 影响范围

- 影响端：Web 管理端。
- 影响页面：瓷砖 SKU、瓷砖品牌、瓷砖类目、瓷砖规格、Banner 管理、用户管理、日志审计、接口文档。
- 主要影响管理端页面一致性、筛选操作效率、表格操作列可达性和分页体验。
- 不影响店主 Web 展示端。
- 不影响微信小程序。
- 当前判断不涉及后端接口契约、数据库结构、MinIO、媒体上传或对象存储策略。
- 当前判断不需要重新生成 Orval，除非后续修复阶段选择调整接口分页结构。

## 严重等级说明

严重等级为 `medium`。

理由：

- 该问题不阻断管理员进入页面，也不造成数据丢失、权限绕过或接口不可用。
- 该问题横跨多个高频管理页面，会持续影响后台信息维护效率和视觉一致性。
- 表格末列未统一固定浮动会影响宽表格场景下行内操作的可达性，属于管理端列表页体验偏差。
- 分页与筛选行为不统一会增加操作学习成本，并影响后续列表页复用规范。
