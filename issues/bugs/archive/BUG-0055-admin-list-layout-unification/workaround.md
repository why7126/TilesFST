# 临时规避方案

## 当前可用规避

在正式修复前，管理员可以继续使用各页面现有筛选、重置、分页和行内操作能力：

1. 对仍有【查询】按钮的页面，先调整筛选条件，再点击【查询】刷新列表。
2. 对已采用即时筛选或无查询按钮的页面，直接修改筛选条件并观察列表变化。
3. 对宽表格页面，如操作列未固定，可横向滚动到表格末尾后执行编辑、启停、删除、查看等行内操作。
4. 对分页只显示当前页的页面，可通过上一页、下一页和每页显示条数完成基础翻页。
5. 若需要快速定位记录，可优先使用关键词筛选缩小结果集。

## 局限

- 规避方式不能消除跨页面操作差异。
- 管理员仍需记忆不同页面的筛选触发方式。
- 宽表格操作列未固定时，行内操作仍可能需要反复横向滚动。
- 当前只有上一页/当前页/下一页的分页控件不能满足“最多 5 个页码可点击”的跳转效率。
- Banner 等页面的模块顺序差异仍会影响视觉扫描路径。

## 是否需要临时开关

不需要。

该问题不涉及数据损坏、权限绕过、上传失败或生产安全风险，不建议通过临时配置隐藏页面或禁用功能。

## Hotfix 判断

不建议 hotfix。

理由：

- 该问题属于 Web 管理端 UI/交互一致性缺陷，不阻断核心业务数据维护。
- 修复涉及多个页面，适合通过常规 `fix-admin-list-layout-unification` OpenSpec Change 统一处理。
- 为避免再次局部修复导致风格分叉，应在修复 Change 中同时覆盖代码、样式和前端回归测试。
