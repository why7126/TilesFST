# 现象

生产环境切换界面主题后，页面提示：

```text
主题已在本机生效，但账号偏好同步失败，请稍后重试
```

提示出现后持续存在，不会自动消失。

# 复现步骤

1. 登录生产环境 Web 管理端。
2. 在登录页、管理端 Shell 侧边栏或其他主题切换入口切换界面主题。
3. 观察是否出现账号偏好同步失败提示。
4. 停留 3 秒以上或继续操作页面，确认提示是否自动消失。
5. 打开浏览器 Network，查看 `PATCH /api/v1/auth/me/theme` 的请求状态码、响应体和耗时。

# 期望结果

- 主题切换后，本机主题应立即生效。
- 已登录用户的账号主题偏好应通过 `PATCH /api/v1/auth/me/theme` 成功保存。
- 如果生产同步失败，应展示可自动消失或可关闭的错误提示，不应持续占据页面。
- 提示文案应明确表达本机主题已生效、账号偏好暂未同步，并允许用户稍后重试。

# 实际结果

- 本机主题切换后出现账号偏好同步失败提示。
- 该提示不会自动消失，持续停留在页面上。
- 当前缺少生产环境 Network 响应体和后端日志，尚不能确认同步失败的具体 HTTP 状态与后端原因。

# 影响范围

- 影响生产环境 Web 管理端主题切换体验。
- 已登录用户的主题偏好可能无法保存到账号，导致刷新、重新登录或跨设备访问时主题不一致。
- 常驻 Toast 会干扰后续操作反馈，用户可能误以为系统仍处于失败或异常状态。
- 若根因是生产 API、鉴权或数据库迁移问题，可能影响所有需要账号偏好同步的用户。

# 严重等级说明

严重等级：`medium`。

理由：该问题不阻断本机主题即时切换，也不直接影响核心瓷砖资料管理流程；但发生在生产环境，且提示常驻与账号偏好同步失败会影响用户信任、跨会话一致性和管理端整体体验。

# 初步分析

前端常驻提示的直接原因倾向于 Toast 生命周期缺失：

- `ThemeContext` 在账号偏好同步失败时设置错误文案，并保留本机主题生效。
- `AdminLayout` 监听到主题错误后写入本地 `toast` state，并清空 ThemeContext error。
- `AdminLayout` 对普通 `notice` 有 3200ms 自动清除逻辑，但对 `toast` 没有自动清除逻辑。
- `AdminToast` 是纯展示组件，不负责倒计时关闭或手动关闭。

账号偏好同步失败的生产原因仍需补充证据：

- 检查 `PATCH /api/v1/auth/me/theme` 是否返回 401、403、404、422、500 或非标准响应 envelope。
- 检查生产用户 Token 是否有效，Authorization Header 是否正常携带。
- 检查生产后端是否部署了主题偏好接口。
- 检查生产数据库 `users.theme_mode` 字段和迁移是否已生效。
- 检查 Web Nginx `/api/` 反代是否正确转发到后端服务。

# 回归关注点

- 主题同步接口成功时，刷新页面或重新登录后仍保持账号偏好主题。
- 同步失败时，本机主题仍生效，错误提示自动消失或可关闭。
- 多次快速切换主题不会产生堆叠、常驻或重复错误提示。
- 未登录用户切换主题不应调用账号偏好同步接口。
- 401/403 等鉴权失败场景不应让用户停留在无法关闭的错误提示中。
