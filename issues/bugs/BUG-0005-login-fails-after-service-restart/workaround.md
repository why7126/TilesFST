# 临时规避方案

## 1. 操作规避

当前没有稳定的产品内规避方案可以在无法登录的情况下自助恢复默认管理员密码。

在正式修复前，本地开发或演示环境可采用以下方式降低影响：

1. 确认根目录 `.env` 中是否配置 `ADMIN_INITIAL_PASSWORD`，且该值是否为首次初始化数据库时使用的密码。
2. 如果仍有其他 `admin` 角色账号可登录，可进入管理端用户管理，通过重置密码能力为目标管理员账号生成新密码。
3. 如果是可丢弃的本地开发数据，可在备份 `data/sqlite/` 后重建本地 SQLite 数据库，并使用明确配置的 `ADMIN_INITIAL_PASSWORD` 完成首次初始化。
4. 如果数据库中包含需要保留的数据，不建议直接删除运行时数据库，应等待修复 Change 提供受控的密码恢复或重置方案。

## 2. 验收规避

在正式修复前，登录验收应区分两类场景：

- 首次启动空数据库：验证 `ADMIN_INITIAL_PASSWORD` 能创建 `admin` 并成功登录。
- 已存在运行时数据库：验证重启后不会误判为“默认密码必然可用”，并记录当前数据库中 admin 密码来源。

## 3. 风险说明

该规避不能消除以下风险：

- 删除或重建 SQLite 数据库会丢失本地运行时业务数据。
- 直接修改数据库密码哈希容易绕过审计和安全流程，不应作为常规操作。
- 若根目录 `.env` 与后端示例环境变量不一致，后续重启仍可能再次出现同类误判。

因此建议进入 `/bug-review`，评审通过后通过 `fix-*` OpenSpec Change 修复默认管理员初始化与恢复流程。
