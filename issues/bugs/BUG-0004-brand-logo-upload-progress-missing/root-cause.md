# 根因分析

## 1. 直接原因

### 1.1 更换 Logo 后缺少可感知的上传状态

用户在编辑品牌弹窗中点击「更换 Logo」并选择图片后，界面没有展示上传进度条、百分比、上传中状态或禁用态。即使底层上传请求已发起，用户也无法从 UI 上判断：

- 是否已开始上传。
- 当前上传进度如何。
- 是否仍需等待。
- 上传成功还是失败。

这使文件选择完成后的等待过程变成“无反馈黑盒”。

### 1.2 上传成功后预览状态未及时更新

缺陷现象显示选择图片后 Logo 预览没有变化。直接原因可能包括：

- 文件选择事件没有触发上传处理逻辑。
- 上传 Promise 尚未完成时没有任何 loading/progress UI。
- 上传成功后未把返回的 `url` / `object_key` 写回弹窗状态。
- `<input type="file">` 选择同一文件时未重置 value，导致再次选择无法触发 `change`。
- 图片预览仅依赖服务端返回 URL，缺少本地临时预览或 cache-busting 机制，导致用户看不到即时变化。

上述原因需在后续修复阶段结合代码与浏览器网络请求确认。

## 2. 根本原因

### 2.1 Logo 上传交互缺少明确的状态机

品牌 Logo 控件目前缺少完整上传状态表达。理想状态机应至少包含：

```text
idle → selecting → uploading(progress) → uploaded / failed
```

当前缺陷说明上传控件只表现为“选择文件入口 + 预览区域”，缺少 `uploading`、`uploaded`、`failed` 等中间态和错误态，导致用户无法理解操作是否生效。

### 2.2 上传进度能力未进入验收闭环

此前品牌 Logo 修复重点在“上传后 URL 可访问、列表/弹窗可展示”，没有把“用户选择文件后必须有上传进度反馈”作为明确验收项。因此即使媒体 URL 可访问，编辑弹窗内的上传体验仍可能不完整。

### 2.3 预览更新与保存链路耦合不清

Logo 更换流程同时涉及：

- 上传文件到后端。
- 获取 `object_key` 和可访问 `url`。
- 更新弹窗内预览。
- 保存品牌时提交新的 `logo_object_key`。
- 重新打开弹窗时回显最新 Logo。

若这些步骤缺少统一状态管理，容易出现“上传未反馈、预览不更新、保存 payload 仍为旧 Logo”的体验缺陷。

## 3. 触发条件

满足以下条件时触发：

1. 用户进入 Web 管理端品牌管理页。
2. 用户打开已有品牌编辑弹窗。
3. 用户点击「更换 Logo」并在系统文件选择窗口选择图片。
4. 文件选择窗口关闭。
5. 弹窗内没有上传进度反馈，且 Logo 预览没有更新。

## 4. 分类结论

| 维度 | 结论 |
|---|---|
| 缺陷分类 | frontend-upload / preview-state / progress-feedback |
| 是否接口缺陷 | 待确认；现象主要表现为前端上传触发与反馈缺失 |
| 是否数据库缺陷 | 否，当前未发现数据库结构问题 |
| 是否权限缺陷 | 待修复阶段回归；上传权限仍需保持 admin / employee 边界 |
| 是否对象存储缺陷 | 待确认；需验证上传请求是否实际到达后端与媒体 URL 是否返回 |
| 是否 Design System 缺陷 | 是，上传控件缺少符合管理端体验的进度反馈 |
| 主要修复面 | 品牌编辑弹窗 Logo 上传状态机、进度条、预览更新、失败/重试反馈 |

## 5. 后续修复建议

1. 为品牌 Logo 上传控件增加明确状态机：空态、上传中、成功、失败。
2. 选择图片后立即触发上传，并在上传过程中展示进度条或等价进度反馈。
3. 上传成功后同步更新 `logoKey`、`logoUrl` 和弹窗预览。
4. 上传失败时展示错误文案，恢复可重试入口，不应静默失败。
5. 文件 input 在处理完成后应允许用户重新选择同一文件。
6. 若 Axios / API 封装支持 `onUploadProgress`，应接入真实进度；若当前封装不支持，应至少提供确定性上传中状态，并在 OpenSpec 修复阶段评估是否扩展上传 API 客户端。
