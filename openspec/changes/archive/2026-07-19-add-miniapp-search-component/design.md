## Context

`REQ-0046-search-component-application` 已评审通过并纳入 `sprint-008`。小程序已有首页、分类列表页、SKU 详情页和搜索入口相关能力，但搜索体验还没有独立规格：搜索入口、搜索首页、实时联想、完整结果、筛选、无结果和搜索埋点容易散落在页面内重复实现。

当前正式范围仅为微信小程序端。评审结论明确排除 Web 管理后台、店主 Web、后台搜索配置中心、管理端搜索配置接口、大模型自然语言搜索、购物车、询价、在线下单、客服找砖或联系商家入口。

## Goals / Non-Goals

**Goals:**

- 沉淀小程序可复用搜索入口组件，支持首页、分类页、SKU 列表、品牌相关页面和独立搜索页复用。
- 提供搜索首页、实时联想、搜索结果、筛选抽屉、无结果和搜索埋点的端到端规格。
- 复用现有公开商品、品牌、类目、规格、证书和媒体安全 URL 边界，不暴露后台内部字段。
- 将小程序搜索事件加入 `product-usage-logging` 事件字典，埋点失败不得阻断搜索主流程。
- 保持 sprint-008 容量风险可控：先完成 `/req-opsx` 规格收敛，再由 `/opsx-apply` 分层实现。

**Non-Goals:**

- 不新增 Web 管理后台搜索组件或管理端列表搜索应用。
- 不新增店主 Web 搜索组件。
- 不新增后台搜索配置中心、热门词维护、同义词维护、自然语言词典维护和搜索统计管理页。
- 不新增 `/api/admin/search/*`。
- 不调用大模型做自然语言解析。
- 不提供购物车、询价、在线下单、客服找砖或联系商家入口。

## Decisions

### D1. 建立独立 `miniapp-search` capability

将小程序搜索定义为独立 capability，而不是继续塞入 `miniapp-home`。原因是搜索入口会被首页、分类页、SKU 列表和品牌相关页面复用，搜索首页和结果页也具备独立状态机、接口和验收基准。

备选方案：修改 `miniapp-home` 的“搜索页与商品详情闭环”。该方案会让首页规格继续膨胀，并难以表达分类页和独立搜索页的复用契约，因此不采用。

### D2. 搜索能力按组件、页面、接口、埋点分层

实现阶段应分为：

1. 小程序搜索入口组件和页面路由。
2. 搜索首页本地状态：最近搜索、热门搜索、最近浏览。
3. 实时联想和完整搜索接口 contract。
4. 结果 Tab、SKU 卡、筛选抽屉和无结果状态。
5. 搜索行为埋点和敏感信息过滤。
6. 原型视觉与触控验收。

这样可以控制 `sprint-008` 已超容量后的回归面，避免把管理端搜索配置中心或完整搜索治理平台带入本 Change。

### D3. 原型冲突处理

本 Change 的 UI 验收优先级为：

```text
prototype/*.html > prototype/*.png > prototype/context.md > acceptance.md > rules/ui-design.md > openspec/specs
```

冲突处理：

- `prototype/context.md` 中的 390px 画布、44px 搜索框、顶部安全区、横向 Tab、72% 筛选抽屉和 62% 遮罩为强约束。
- 原型中的证书只允许出现在完整结果，不得出现在联想。
- SKU 卡片整卡进入详情，不显示收藏/分享快捷操作。
- 若既有 `miniapp-home` 仍描述“搜索页与商品详情闭环”，以本 Change 对搜索组件和搜索页的更细规格为准。

### D4. API 和数据边界

可以复用已有公开商品、品牌、类目、规格和证书查询能力；若复用不足，可新增小程序搜索相关接口，例如联想、完整搜索和热门词。任何新增或调整 API 必须同步 OpenAPI、Orval、docs 和测试。

搜索响应必须二次过滤公开状态：下架 SKU、停用品牌、停用类目、停用规格和不可公开证书不得出现在联想、结果、聚合数量或热门统计中。

### D5. 埋点扩展放入 `product-usage-logging`

小程序搜索事件属于产品使用行为事件，应扩展既有事件字典，而不是新建独立日志能力。事件参数只能包含关键词、归一化关键词、范围、实体类型、结果数量、来源页面、筛选快照和 requestId 等必要字段，不得包含手机号、Authorization header、Cookie、聊天内容或 raw payload。

## Risks / Trade-offs

- **容量超载** → `sprint-008` 已达到 31.0 / 30.0 人天；实现阶段必须冻结范围外扩，先完成 Change 规格与核心搜索路径。
- **管理端配置范围外溢** → proposal、design、spec 和 tasks 均明确不做后台搜索配置中心和 `/api/admin/search/*`。
- **搜索接口性能不达标** → tasks 必须覆盖联想防抖、请求顺序、分页和接口测试；必要时先采用参数化数据库查询和有限字段返回。
- **敏感信息误入埋点** → specs 明确禁止敏感字段；后端测试覆盖禁止属性和埋点失败不阻断。
- **原型和实现不一致** → 实现阶段必须以 HTML/PNG/context 为视觉优先级，并补小程序触控与视口验收证据。

## Migration Plan

1. `/opsx-apply add-miniapp-search-component` 前确认 `REQ-0046` 与 `sprint-008` 双向追溯完整。
2. 先实现或确认小程序搜索 API contract，再接入小程序页面状态。
3. 引入小程序搜索组件和页面后，保留旧首页/分类页入口可安全跳转。
4. 搜索埋点新增后，失败只记录调试信息，不阻断用户搜索流程。
5. 若发现容量无法承载完整范围，优先保留搜索入口、联想、结果和无结果，筛选增强或埋点细项可拆分后续 Change。

## Open Questions

- 热门搜索词 v1 是否完全使用静态配置/系统热度补足，还是复用已有公开数据排序字段，需在 `/opsx-apply` 设计实现时确认。
- 品牌页是否已有稳定路由；若没有，品牌建议可先进入搜索结果页或安全降级。
