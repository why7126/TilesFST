# Design - fix-miniapp-search-prototype-alignment

## Bug Analysis Report

| 项 | 结论 |
|---|---|
| 现象 | 小程序搜索页的首页、联想、结果、筛选和无结果状态与 REQ-0046 原型不一致 |
| 严重等级 | high |
| 影响终端 | 微信小程序 |
| 关联需求 | `REQ-0046-search-component-application` |
| 来源 Change | `add-miniapp-search-component` |
| 主要风险 | 原型验收误判、搜索体验不一致、后续扩展返工 |

## Root Cause

实现阶段完成了搜索数据流和基础页面骨架，但缺少“原型结构级”验收门禁：

- 搜索页顶部结构使用 `custom-navigation` 加自建搜索区域，未按原型的一体化返回按钮、搜索框、取消/搜索按钮落地。
- 独立搜索页未复用已创建的 `components/search-entry`，导致通用组件行为和页面实现脱节。
- 页面 data 已承载 `SearchResponse.sections`，但 WXML 未按综合分区渲染。
- 结果页不再展示快捷筛选、筛选按钮或筛选抽屉，避免搜索结果页承担二次筛选操作。
- 无结果状态缺少搜索图标、结构化调整建议和推荐搜索区域。
- 现有静态测试未断言原型关键结构，导致任务完成状态与实际页面偏差不一致。

## Fix Strategy

### 1. 搜索页顶部和通用组件应用

- 独立搜索页顶部必须按原型结构呈现返回按钮、搜索框、取消/搜索按钮，并位于顶部安全区下方。
- 优先复用 `components/search-entry`；若因小程序页面结构限制无法直接复用，页面实现必须保持该组件等价行为，并在测试中覆盖等价性。
- 保持关键词、清空、提交、取消/返回、禁用态、`scope` 和 `sourcePage` 行为一致。

### 2. 五个原型状态修复

- 搜索首页：展示最近搜索和热门搜索，不展示最近浏览；最近搜索支持单条删除和全部清空，清空按钮与搜索框右侧对齐。
- 联想态：仅展示品牌和 SKU 两组，不展示最近搜索、普通关键词、类目、规格或证书。
- 结果态：综合、品牌、SKU、证书 Tab 横向滚动；综合默认激活；综合页渲染最多 1 条最佳匹配，并按品牌、SKU、证书顺序展示非 0 条分区；单独 Tab 不展示分区标题和数量。
- 最佳匹配：SKU 编码或名称直接命中优先，其次品牌名精确命中，最后证书名称或证书编号精确命中。
- 无结果态：展示当前关键词、搜索图标、调整建议列表和推荐搜索词，不展示联系商家、找砖、购物车、询价或在线下单入口。

### 3. 测试与验收门禁

- 补充小程序静态或单元测试，覆盖 WXML 结构、事件名称、筛选入口、价格区间、`sections` 分区和无结果结构。
- 补充原型对照证据，至少覆盖 `01-search-home` 至 `05-search-empty`。
- 修复完成前不得把任务状态标记为完成，除非测试和原型对照均已记录。

## API / DB Decision

本 BUG 的当前证据指向小程序页面结构、样式和测试缺口，不要求默认修改 API 或数据库。实现中若确认现有 API 无法提供综合分区、facet 数量、价格区间或应用筛选后数量，必须先更新本 Change 的 delta spec、API 文档、OpenAPI/Orval 和接口测试，再继续实现。

## Regression Scope

- 小程序搜索入口组件。
- 独立搜索页 5 个状态。
- 搜索结果综合分区和 SKU 卡片点击。
- 结果页无筛选 UI；品牌、SKU、证书结果卡片点击行为需保持各自目标路径。
- 无结果、加载、失败、重试与状态切换稳定性。
