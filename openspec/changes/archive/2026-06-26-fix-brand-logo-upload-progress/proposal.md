## Why

`BUG-0004-brand-logo-upload-progress-missing` 已评审通过并纳入 `sprint-002`。当前 Web 管理端编辑品牌弹窗中，用户点击「更换 Logo」并选择图片后，没有明显上传反馈，Logo 预览也没有更新。用户无法判断图片是否正在上传、上传是否成功，或需要等待多久。

该缺陷影响 `REQ-0005-brand-management` 的品牌 Logo 维护体验，并与已归档的 `BUG-0003-brand-image-display-layout-shift` 后续链路相关：媒体 URL 已可访问，但编辑弹窗内的“选择文件 → 上传中 → 成功预览 / 失败重试”交互闭环仍不完整。

## What Changes

- 修复 `BrandFormModal` 的 Logo 更换流程：
  - 选择文件后立即触发上传。
  - 上传过程中展示进度条、百分比或等价可感知反馈。
  - 上传成功后更新弹窗 Logo 预览和待保存的 `logo_object_key`。
  - 上传失败时展示错误并保留重试入口。
  - 支持重新选择同一文件进行重试。
- 补充前端回归测试，覆盖上传中状态、成功预览、失败错误和重试。
- 若为真实上传进度需要调整前端 API 封装，需同步相关类型、调用方和必要测试。

## Impact

| 影响面 | 说明 |
|---|---|
| Web 管理端 | 修改品牌编辑弹窗 Logo 上传交互和状态反馈 |
| 后端 API | 默认不新增端点、不变更响应 schema；若实现真实进度回调仅调整前端请求封装 |
| 数据库 | 不变更 |
| Orval | 默认不需要；若 API schema 变更则必须重新生成 |
| 小程序 | 不涉及 |
| 测试 | 需补充或更新 Vitest；必要时回归后端上传测试 |
| Design System | 进度条、按钮禁用态、错误文案需使用管理端样式变量或语义 Token |

## Rollback Plan

如修复引起品牌弹窗上传流程异常，可回滚本 change 涉及的前端代码改动：

1. 恢复 `BrandFormModal` Logo 上传控件到修复前状态。
2. 恢复相关品牌上传 API 调用封装（若有调整）。
3. 保留现有品牌 Logo 可访问媒体链路，不回滚 `BUG-0003` 的后端 `/media` 修复。
4. 回滚后重新标记 `BUG-0004` 未修复，并保留验收失败记录。
