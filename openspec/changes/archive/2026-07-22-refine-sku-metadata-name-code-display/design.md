## Context

REQ-0065 has approved a refinement to SKU metadata semantics: SKU code remains the unique internal identifier, while product name is the user-entered and publicly displayed field. Current specs and implementation still treat `sku_code` as an operator-entered create field and public display field in miniapp detail/list/search/share surfaces.

Existing constraints:

- `tiles.sku_code` exists as `UNIQUE NOT NULL` and must remain stable for logs, API compatibility, migration and internal lookup.
- Admin SKU management already has a wide modal and list page; the change should preserve existing admin-list and admin-modal best-practice gates.
- Miniapp product detail, product list and search specs currently allow or require displaying SKU code. This change updates public display semantics without creating a new public capability.

## Goals / Non-Goals

**Goals:**

- Generate SKU code server-side for draft and create flows.
- Keep SKU code unique, stable and internally searchable.
- Use product name as the main admin and public display label.
- Hide SKU code from miniapp/storefront cards, detail parameters, recommendations, favorites, search results and share titles.
- Keep API, OpenAPI, Orval, docs and tests synchronized when request/response contracts change.
- Preserve `admin-list` and `admin-modal` cross-cutting gates from the REQ knowledge-base refs.

**Non-Goals:**

- Do not remove `tiles.sku_code`.
- Do not introduce manual code-rule configuration, ERP code sync, bulk import/export rules, aliases or multilingual names.
- Do not change media upload, object storage, price, brand, category, spec or status capabilities except where labels/search reference name/code.

## Decisions

### D1. UI Strategy: Targeted DS/CSS Port Refinement

Use the existing SKU management admin page and `TileSkuFormModal` CSS port strategy. The prototype is a field-semantics reference, not a new page layout. Admin implementation must keep existing `sku-modal-card` ownership and must not introduce `modal-card` double mounting.

Alternatives considered:

- Full redesign of SKU management page: rejected because REQ-0065 only changes metadata semantics and public display.
- New shared SKU name/code component: deferred until duplication appears across more admin pages.

### D2. Code Generation: Backend Owns SKU Code

The backend service owns SKU code generation for both draft and create flows. Frontend create/update forms should not require operators to type `sku_code`. Generated codes must not derive from product name in a way that changes when the name changes.

Alternatives considered:

- Keep user-entered code but hide it publicly: rejected because it does not reduce operator duplication.
- Generate code on the frontend: rejected because uniqueness and collision handling belong in backend/domain logic.

### D3. API Compatibility

Admin create request should stop requiring `sku_code`; response may continue returning `sku_code` for admin/internal compatibility. Miniapp/storefront responses may keep `sku_code` as a compatibility field if already generated by shared schemas, but public clients must not render it.

If removing `sku_code` from any request schema or generated client type, update OpenAPI, Orval, API docs and tests in the same apply.

### D4. Public Display Contract

Miniapp and storefront surfaces use product name as the public SKU label. SKU code may still be accepted as a backend search signal, but search result cards and detail pages must not show it.

### D5. Migration

Existing SKU codes remain unchanged. Missing historical SKU codes, if any, must be backfilled with unique generated codes before records can be used in create/update/public flows. Historical name/code similarity is not auto-cleaned by this change.

## Conflict Resolution

Priority order: prototype HTML > prototype context > acceptance > UI design rules > existing specs.

| Conflict | Resolution |
|---|---|
| Existing `tile-sku-management` spec says create requires SKU code | Update spec so backend generates code and frontend does not require input |
| Existing miniapp SKU detail spec says display SKU code | Update detail display, parameters and share scenarios to hide code |
| Existing product list spec allows card SKU code display and requires response include code | Update public display to hide code; response may keep internal compatibility field |
| Existing search spec prioritizes SKU code and no-result advice mentions encoding | Keep backend matching compatibility, but public result and advice copy must not display code |

## Risks / Trade-offs

- [Risk] Hidden `sku_code` still exists in public API response and could be accidentally rendered later → Mitigation: add frontend/static tests asserting no SKU code text appears in public surfaces.
- [Risk] Generated code collision under concurrent creates → Mitigation: preserve DB unique constraint and add retry/error handling tests.
- [Risk] Orval generated client drifts from backend if create schema changes → Mitigation: make OpenAPI/Orval generation and generated type diff part of tasks.
- [Risk] Admin users lose exact lookup affordance → Mitigation: keep admin keyword search by product name or SKU code and optionally display code as weak internal info.

## Migration Plan

1. Preserve `tiles.sku_code` and existing values.
2. Adjust backend create/update validation so `sku_code` is generated on create and stable on update.
3. Add or reuse a backfill helper for any missing legacy `sku_code` records if schema/history requires it.
4. Update admin UI and public miniapp/storefront rendering.
5. Regenerate OpenAPI/Orval if request/response schemas change.
6. Run backend, web and miniapp focused tests.

Rollback: since the DB field remains, rollback can restore prior UI/request behavior without data loss. Generated codes created during this change remain valid SKU codes.

## Open Questions

- Whether admin detail should always display SKU code as weak internal text or expose it only in copy/debug affordances can be decided during implementation, as long as product name remains primary.
