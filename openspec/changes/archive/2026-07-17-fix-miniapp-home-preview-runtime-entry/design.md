# Design: 小程序首页预览运行入口修复

## 1. 背景

`add-miniapp-home` 已实现小程序首页相关 API、页面和测试，但用户在微信开发者工具中预览首页时，看到的结果与原型明显不一致。BUG-0065 的根因分析指出：页面 WXML/WXSS 和 `.ts` 业务逻辑存在，但同目录 `.js` 仍是空模板，真实预览未执行 `loadHome()`。

## 2. 根因映射

| 根因 | 设计响应 |
|---|---|
| `.ts` 与 `.js` 事实源脱节 | 明确运行事实源，确保微信开发者工具加载的 `.js` 包含业务逻辑，或建立可验证的 TS 编译链 |
| 空模板 `.js` 未被测试捕获 | 增加静态测试检查关键页面运行脚本不得为空模板 |
| 首页预览未触发接口请求 | 修复后通过手工预览或等价 smoke 验证 `GET /api/v1/miniapp/home` 发起 |
| 容易误判为后端数据缺失 | 验收区分“运行入口执行”“接口可访问”“无数据模块级降级”三类证据 |

## 3. 修复方案

### 3.1 运行入口策略

实现阶段必须选择一种明确策略：

1. **同步运行脚本策略**：将关键页面业务逻辑同步到微信开发者工具实际加载的 `.js` 文件，并保持 `.ts`/`.js` 不脱节。
2. **可验证 TS 编译链策略**：配置项目使用 TypeScript 编译链生成 `.js`，并确保生成产物被微信开发者工具加载。

无论采用哪种策略，验收必须证明：

- `pages/index/index` 实际运行脚本包含 `data.loading`、`home`、`loadHome()`。
- `onLoad()` 会触发首页数据加载。
- 关键页面不再保留空模板 `Page({ data: {}, onLoad() {} })`。

### 3.2 首页首屏修复边界

修复后首页应恢复既有设计目标：

- 品牌导航、搜索入口、Banner、快捷入口和至少一个推荐模块可见。
- 无公开商品时，推荐商品模块可隐藏或显示品牌化空态，但 Banner、快捷入口、服务区不能整体消失。
- Banner 或推荐商品图片失败时，应模块级降级，不阻塞其他模块。

### 3.3 测试策略

新增或调整测试时，至少覆盖：

- 小程序首页运行脚本不是空模板。
- 首页运行脚本包含关键数据字段、接口调用或 `loadHome` 行为。
- 搜索页、商品详情页、门店信息页等关键页面不得保留空模板运行脚本。
- 既有后端 `GET /api/v1/miniapp/home` 测试继续通过。

### 3.4 范围控制

本 Change 默认不改 API、DB、Orval、Docker Compose。若实现过程中发现接口契约必须改变，需要在本 Change 中补充：

- API spec delta。
- API 文档。
- OpenAPI / Orval 生成结果。
- 后端集成测试。

## 4. 验收与证据

实现完成后应记录：

- 微信开发者工具首页预览截图或等价说明。
- 375x812、390x844 和 320 到 430 pt 范围验证结果。
- 自动化测试命令和结果。
- API/DB/Orval/Docker 是否变更的范围检查结论。

## 5. Knowledge Base

本缺陷具有复用价值：它暴露出“小程序页面源码与运行入口脱节”的工程风险。修复完成后，如验收确认该问题可能复现于其他页面，应沉淀到 `docs/knowledge-base/incidents/`。
