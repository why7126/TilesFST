## Context

`BUG-0079` 指出管理端首页数据概览仍展示 Mock 数据。当前正式规格 `openspec/specs/admin-dashboard/spec.md` 的「管理端 Dashboard 数据概览」也保留了“本期 MUST 使用 mock 数据”的历史约束，因此修复既需要实现层替换数据源，也需要规格层更新验收口径。

该问题属于 code / product-quality 类缺陷。它不阻断管理端登录或 CRUD，但会让首页关键指标失真，影响管理员对业务规模和运营状态的判断。

## Goals

- 管理端 `/admin/dashboard` 数据概览显示真实业务数据。
- 保留现有 4 个指标卡：SKU 总数、品牌数量、Banner 数量、用户数量。
- 统计口径与后端查询或对应管理端列表页一致。
- 请求中、空数据、错误和无权限场景均有明确 UI 状态，不使用 Mock 数据伪装成功结果。
- 真实数据接入遵守管理端鉴权与权限边界。
- 若涉及 API 变更，同步 OpenAPI、Orval、API 文档与测试。

## Non-Goals

- 不新增 Dashboard 新指标或图表。
- 不改变管理端 Shell、Sidebar、快捷操作、最近更新表格和视觉主题。
- 不改变用户角色模型、登录流程或权限体系。
- 不新增数据库表；统计应基于现有业务表。
- 不实现完整运营分析看板或历史趋势分析。

## Root Cause

直接原因是管理端首页数据概览仍从 Mock 常量、演示数据或临时 fallback 数据源读取指标。

根本原因是 Dashboard 从原型阶段进入正式页面后，缺少“Mock 数据退出”门禁；规格本身也继续允许 Mock 数据，使实现和验收都没有强制真实数据源闭环。

## Proposed Design

1. 数据源策略：
   - 优先复用现有后端管理端 API 或新增受鉴权保护的 dashboard summary API。
   - 指标卡不得直接读取页面硬编码 Mock 常量作为成功态数据。
   - 测试夹具、Storybook 或本地组件测试可以保留 Mock，但必须与生产页面运行路径隔离。
2. 统计口径：
   - SKU 总数应与 SKU 管理列表或后端 SKU 查询总量一致。
   - 品牌数量应与品牌管理列表或后端品牌查询总量一致。
   - Banner 数量应与 Banner 管理列表或后端 Banner 查询总量一致。
   - 用户数量应与用户管理列表或后端用户查询总量一致，并遵守可见权限。
3. UI 状态：
   - loading：首次请求中显示加载态或骨架态。
   - empty：真实统计为 0 时展示 0 或明确空状态。
   - error：接口失败、网络异常或鉴权失败时展示错误状态或重试入口，不展示 Mock 值。
4. API 与权限：
   - 若新增接口，路径必须使用 `/api/v1` 前缀，响应使用统一 envelope。
   - 接口必须要求管理端鉴权，并遵守 admin / employee 数据可见边界。
   - 错误响应不得泄露数据库 DSN、SQL、MinIO 凭据、内部堆栈或密钥。
5. 回归边界：
   - 修复不得改变 Dashboard 四卡布局、品牌金数值强调和桌面四列网格。
   - 不影响 Sidebar、快捷操作、最近更新和已删除模块约束。

## Testing

- 后端/API 测试覆盖真实统计成功返回、空数据返回 0、未授权访问失败、至少一个指标与业务表数量一致。
- 前端 Vitest / Testing Library 覆盖 Dashboard 数据概览调用真实 API 客户端、渲染真实数值、loading、empty 和 error 状态。
- 若新增或修改 API，运行 OpenAPI / Orval 生成并回归使用生成客户端的前端测试。
- 如具备本地环境，使用 Docker Compose 或等价本地服务完成管理端首页 smoke 验证。

## Risks

- 各业务列表页统计口径可能不一致，导致首页指标与列表页总数存在差异。
- 直接串行请求多个列表接口可能增加首页加载时间。
- employee 权限边界若未定义清楚，用户数量或系统级指标可能存在越权展示风险。
- 接口失败时从 Mock 兜底切换为错误状态，可能暴露此前被隐藏的后端可用性问题。

缓解方式：以 AC-001 至 AC-007 为验收标准，优先定义统一 dashboard summary 响应或统一统计口径，并补充自动化回归。
