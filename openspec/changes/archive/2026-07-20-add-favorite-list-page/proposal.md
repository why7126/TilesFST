## Why

用户在选砖过程中会反复比较多个瓷砖 SKU、品牌或内容对象。当前展示端和小程序已有首页、分类、搜索、商品列表和 SKU 详情等浏览链路，但缺少统一的收藏列表页，用户只能依赖重复搜索、浏览路径回溯或外部截图记录候选对象，回访和对比效率低。

REQ-0059 已评审通过，需要将“新增收藏列表页”转为 OpenSpec Change，明确用户侧收藏列表的页面、状态、操作、数据一致性和验收边界，为后续实现提供规范事实源。

## What Changes

- 新增用户侧收藏列表页能力，集中展示当前用户已收藏对象。
- 首期建议优先支持 SKU / 商品收藏，其他对象类型作为可扩展能力进入设计边界。
- 收藏列表支持进入详情、取消收藏、空状态、未登录状态、异常状态、刷新和增量加载。
- 收藏状态需要与详情页保持一致，避免列表与详情收藏标识不同步。
- 若实现覆盖微信小程序，必须引用小程序自定义导航 best practice，完成状态栏、胶囊 reserve、返回兜底、页面 offset 和设备 evidence 策略。
- 不包含管理端收藏运营、购物车、询价、在线下单、收藏分组、分享收藏夹或推荐算法。

## Capabilities

### New Capabilities

- `favorite-list-page`: 用户侧收藏列表页，支持收藏对象展示、详情跳转、取消收藏、空/错/未登录状态和基础埋点。

### Modified Capabilities

- 无。

## Impact

- 影响端：微信小程序、店主 Web 展示端待实现阶段确认；产品语义为用户侧能力，默认不影响 Web 管理端。
- 可能涉及源码：
  - `src/miniapp/pages/favorites/*`
  - `src/miniapp/app.*` 或页面路由配置
  - `src/miniapp/components/product-card/*` 或等价收藏卡片复用
  - 若覆盖 Web 展示端，可能涉及 `src/web/src/pages/`、`src/web/src/shared/business/` 或用户侧路由
- API：可能需要新增或复用用户收藏查询、取消收藏、状态同步接口；实现阶段必须明确请求/响应与错误码。
- 数据库：若当前无收藏数据模型，可能需要新增用户收藏表；实现阶段必须同步 schema、数据库文档和测试。
- 对象存储：无上传或存储策略变更。
- 管理端：无影响。
- Orval：若新增或修改后端 API，必须同步 OpenAPI / Orval。
- 测试：需要补充前端/小程序静态或单元测试、后端接口测试（若涉及 API/DB）、状态同步与空/错/未登录验收。
