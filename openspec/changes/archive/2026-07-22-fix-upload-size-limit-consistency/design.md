## 背景

`BUG-0073` 记录了约 23M 文件上传失败的问题。用户补充该问题不止视频，图片也存在，文档也预计会受影响。当前代码和配置显示图片默认 20MB、视频默认 500MB、品牌证书 / PDF 硬编码 20MB，上传限制缺少跨类型一致治理。

## 根因

直接原因：

- 图片上传使用 effective `media.max_image_size_mb` / `MAX_IMAGE_SIZE_MB`，默认 20MB。
- SKU 视频上传使用 effective `media.max_video_size_mb` / `MAX_VIDEO_SIZE_MB`，默认 500MB。
- 品牌证书 / PDF 上传使用 `CERTIFICATE_MAX_SIZE_MB = 20`，不受系统设置和环境变量统一控制。
- 不同上传入口对约 23M 文件表现不同，且前端提示、后端校验、部署代理配置和错误信息可能不一致。

根本原因：

- 缺少图片、视频、文档三类上传大小限制的统一产品规格。
- 文件/证书类没有纳入 `EffectiveSettingsService` 媒体配置。
- 回归测试偏向“超限会拒绝”，缺少边界文件成功/清晰拒绝证据。

## 修复方案

1. 明确上传大小策略：
   - 图片、视频、文档三类限制必须有明确 effective 来源。
   - 若产品确认约 23M 图片/文档应允许上传，则默认或系统设置需大于等于 23MB。
   - 若产品确认仍不允许，则前端提示和后端错误必须明确当前限制。
2. 后端统一限制读取：
   - 保留 `media.max_image_size_mb` 与 `media.max_video_size_mb`。
   - 新增或等价实现 `media.max_file_size_mb` / `MAX_FILE_SIZE_MB`，供文档、PDF、证书类上传使用。
   - 品牌证书上传不得继续使用不可配置的 20MB 常量作为唯一事实源。
3. 错误响应与安全：
   - 超限返回统一结构和明确错误码。
   - MIME Type、扩展名、对象 Key 校验、管理端鉴权和 MinIO / S3 适配层必须保留。
   - Nginx / Docker / 生产代理请求体限制必须大于等于后端最大 effective 上传限制，不得让 413 替代业务校验。
4. 前端与设置：
   - 管理端上传入口显示与后端 effective 限制一致的提示。
   - 系统设置 media Tab 如新增文档类限制，需要展示、保存、恢复默认和校验。
5. 文档与客户端：
   - API 字段或响应结构变更时同步 OpenAPI / Orval。
   - `.env.example`、媒体文档、对象存储文档和部署文档同步限制说明。

## 测试策略

- 后端单元 / 集成测试：
  - 图片：约 23M 文件在允许配置下成功；超过上限返回 `FILE_SIZE_EXCEEDED`。
  - 视频：约 23M MP4 在视频入口成功；不支持 MIME 被拒绝。
  - 文档 / 证书：约 23M PDF 在允许配置下成功；超过文件类上限返回清晰错误。
  - 存储不可用：MinIO / S3 写入失败返回统一对象存储不可用错误，不泄露内部路径。
  - 系统设置：PATCH media 分组修改文件类上限后无需重启即生效。
- 前端测试：
  - 上传提示显示 effective 限制。
  - 超限和类型不支持时显示稳定错误状态，可重试。
- 部署验证：
  - Docker Web 入口 `client_max_body_size` 与最大上传限制一致。
  - 生产代理配置不得低于后端最大 effective 上传限制。

## 非目标

- 不新增视频转码、压缩、多清晰度或封面生成能力。
- 不改为前端直传对象存储。
- 不引入多个 Bucket。
- 不提交真实客户文件或大体积测试样本。
- 不绕过管理端鉴权、MIME Type 校验、扩展名校验、对象 Key 校验或对象存储适配层。
