## Context

当前 `tile-category-management` 正式规格和后端异常仍表达“最多三级”，Web 类目表单也提示“最多支持三级类目”。小程序分类列表页已按一级/二级两级结构实现，为保持运营后台与小程序交互一致，需要阻止后台新增三级类目。

## Goals / Non-Goals

**Goals:**

- 后端成为最终约束源：任何创建三级类目的请求都被拒绝。
- Web 管理端提前过滤二级类目，不允许将二级类目作为新类目的上级。
- 类目列表和 summary 对外表达最大层级为 2。
- 保留现有 API 路径、请求体和错误码形态，降低客户端改造面。

**Non-Goals:**

- 不新增数据库迁移，不删除或自动折叠历史三级类目数据。
- 不改变 SKU 绑定类目规则或小程序分类树公开接口。
- 不新增管理端批量迁移、历史数据治理或运营提示面板。

## Decisions

- 后端创建校验使用父类目 `level >= 2` 拦截，沿用 `CategoryMaxDepthExceededError` 和 `CATEGORY_MAX_DEPTH_EXCEEDED`，避免新增错误码。
- 列表接口的 `level` 查询参数限制为 1 或 2；summary `max_level` 固定为 2，以便管理端展示与新业务上限一致。
- Web 表单的上级类目选项只展示一级类目；编辑模式继续不允许变更 `parent_id`，因此不扩大编辑行为。
- 历史三级类目如已存在，树接口仍可返回以避免隐藏存量数据造成运营困惑；本变更重点是禁止新增三级类目。

## Risks / Trade-offs

- 历史三级数据仍可能出现在管理端树中 → 本变更不做破坏性迁移，后续可单独提出历史数据治理 Change。
- SKU 当前若绑定到三级类目，仍保留原绑定 → 避免无授权地改动商品数据；新增类目上限不影响既有 SKU 读取。
- 前端过滤只能改善体验，不能作为唯一约束 → 后端校验作为最终防线。
