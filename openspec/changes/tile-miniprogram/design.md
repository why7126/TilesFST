## Context

本项目是瓷砖信息管理平台的小程序前端部分，目标是为渠道商（零售店老板）提供移动端产品浏览能力。

技术约束：
- 前端使用 TypeScript 开发
- 使用微信小程序开发框架
- 数据存储在本地（Storage），不涉及服务端数据库
- 调用后端 REST API 获取产品数据（tile-api-backend）
- 不直接访问 MinIO，通过后端 API 获取图片/视频访问地址

## Goals / Non-Goals

**Goals:**
- 实现产品列表展示，支持多维筛选和关键词搜索
- 实现产品详情页，支持图片轮播、视频播放、属性展示、价格展示
- 实现本地收藏功能
- 实现本地浏览记录功能
- 实现产品分享功能

**Non-Goals:**
- 用户登录体系（纯公开浏览）
- 实时库存管理
- 微信支付
- 后端数据管理（Web后台能力）

## Decisions

### 1. 技术选型：微信小程序原生开发 vs uni-app

**决策：微信小程序原生开发**

理由：
- 项目规模较小，功能相对简单，原生开发更轻量
- 不需要跨平台（iOS/Android/H5），单一平台足够
- 原生开发调试更方便，性能更好

### 2. 本地存储策略

**决策：使用 wx.setStorage / wx.getStorage 存储收藏和浏览记录**

理由：
- 小程序端原生支持，无需额外依赖
- 数据存储在设备本地，简单可靠
- 不需要登录即可使用

### 3. 数据交互方式

**决策：通过后端 REST API 获取数据，前端不直接访问 MinIO**

理由：
- 符合安全规范，前端不持有 MinIO 凭证
- 后端可以统一处理缓存、权限校验
- 通过预签名 URL 获取图片/视频访问地址

### 4. 搜索实现

**决策：前端维护搜索历史（本地），后端提供全文搜索能力**

理由：
- 搜索历史需要本地存储，使用本地缓存即可
- 后端使用 MySQL 全文索引（tile-api-backend 提供）

## Risks / Trade-offs

| 风险 | 描述 | 缓解方案 |
|-----|------|----------|
| 分享追踪困难 | 不登录无法追踪分享来源 | 仅记录分享事件数量，不追踪具体来源 |
| 数据不跨设备 | 收藏/浏览记录不跟随账号 | 明确告知用户数据存本地 |
| 图片加载性能 | 10000 SKU，每个10张图，加载慢 | 实现图片懒加载，使用 CDN 加速 |
| API 依赖 | 完全依赖后端 API | 做好错误提示和重试机制 |
