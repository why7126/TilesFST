## Context

本项目是瓷砖信息管理平台的 Web 前端部分，目标是为渠道商（零售店老板）提供桌面端产品浏览能力。

技术约束：
- 前端使用 React + TypeScript 开发
- 使用响应式布局，支持桌面端浏览器
- 数据存储在浏览器 localStorage，不涉及服务端数据库
- 调用后端 REST API 获取产品数据（tile-api-backend）
- 不直接访问 MinIO，通过后端 API 获取图片/视频访问地址

## Goals / Non-Goals

**Goals:**
- 实现产品列表展示，支持多维筛选和关键词搜索
- 实现产品详情页，支持图片轮播、视频播放、属性展示、价格展示
- 实现本地收藏功能
- 实现本地浏览记录功能
- 实现产品分享功能（复制链接）

**Non-Goals:**
- 用户登录体系（纯公开浏览）
- 实时库存管理
- 微信支付
- 后端数据管理（Web后台能力）

## Decisions

### 1. 技术选型：React + TypeScript + Vite

**决策：使用 React + TypeScript + Vite**

理由：
- 项目使用 TypeScript，与后端 Python 保持类型一致
- Vite 提供更快的开发启动和热更新
- React 生态成熟，组件库选择丰富

### 2. 状态管理方案

**决策：使用 React Context + localStorage**

理由：
- 项目规模较小，不需要复杂的状态管理
- 使用 localStorage 持久化收藏和浏览记录
- 简单易用，无需额外依赖

### 3. 路由方案

**决策：使用 React Router**

理由：
- 标准的前端路由方案
- 支持懒加载，优化首屏性能
- 与 React 生态集成良好

### 4. 数据交互方式

**决策：通过后端 REST API 获取数据，前端不直接访问 MinIO**

理由：
- 符合安全规范，前端不持有 MinIO 凭证
- 后端可以统一处理缓存、权限校验
- 通过预签名 URL 获取图片/视频访问地址

## Risks / Trade-offs

| 风险 | 描述 | 缓解方案 |
|-----|------|----------|
| 浏览器兼容性 | 不同浏览器可能存在兼容性问题 | 使用标准 Web API，做好降级处理 |
| 数据不跨设备 | 收藏/浏览记录不跟随账号 | 明确告知用户数据存本地 |
| 图片加载性能 | 10000 SKU，每个10张图，加载慢 | 实现图片懒加载，使用 CDN 加速 |
| API 依赖 | 完全依赖后端 API | 做好错误提示和重试机制 |
